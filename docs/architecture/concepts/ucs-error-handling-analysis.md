# UCS Error Handling: Current State & Unification Options

**Purpose:** Document current error handling inconsistencies and unification options
**Date:** 2026-03-30

---

## Summary

UCS (Unified Connector Service) has **two types of errors**:

**1. Connector Business Errors** (card declined, insufficient funds, invalid CVV)
- Generated by connector integrations as `ErrorResponse`
- Should populate the `ErrorInfo` field in response proto, and already done that way for all flows.
- Returns `Ok(Response)` with `ErrorInfo` field populated

**2. UCS Integration Errors** (missing required fields, bad config, proto parsing)
- Infrastructure/integration failures on UCS side
- Should return `Err(tonic::Status)` with appropriate gRPC code
- No response body (request never completed successfully)

**Current Implementation:**

| Flow | Business Errors | Integration Errors |
|------|----------------|-------------------|
| **Authorize** | `ErrorInfo` field ✓ | `ErrorInfo` field ✓ |
| **Capture/Refund/etc** | `ErrorInfo` field ✓ | `Err(tonic::Status)` ✓ |

**The Inconsistency:** Authorize flow converts integration errors to `ErrorInfo` field, while other flows (capture, refund, void, psync) convert integration errors to `Err(tonic::Status)`.

**Downstream Impact:**
- Authorize business errors → HS receives `ExternalConnectorError` with details preserved
- Other flow business errors → HS receives generic `InternalServerError` (details lost)
- HTTP clients see HTTP 200 with error JSON for authorize, but 4xx/5xx for other operations


---

## The Inconsistency

### Root Cause

Two different error handling implementations coexist in UCS:

| Aspect | **Authorize** | **Macro-Generated Flows** |
|--------|--------------|--------------------------|
| Implementation | Hand-written in `payments.rs` | Generated by `implement_connector_operation!` macro |
| Domain error type | `PaymentAuthorizationError` | `ApplicationErrorResponse` |
| Conversion strategy | `From<Error> for Response` | `IntoGrpcStatus` trait |
| Wire format | `Ok(Response)` with error fields | `Err(tonic::Status)` |
| HTTP behavior | Always 200 | Status-dependent (400/500/503) |

### Why This Exists

- Not a bug or oversight
- Authorize was implemented with domain-first error modeling
- Macro flows followed standard gRPC `Err(Status)` pattern
- Both patterns are valid; the inconsistency is about **engineering complexity** and **future maintainability**

**Current impact:** No urgent business issues, but documented for **future planning** as payment flows expand.

## How This Manifests

### 1. Merchant API Perspective

The same class of connector error produces different API responses:

| Operation | UCS Response | HS Behavior | Merchant Sees |
|-----------|-------------|-------------|---------------|
| **Authorize** fails | `Ok(Response)` with `error_code: "MISSING_FIELD"` | `RouterData.response = Err(ErrorResponse)` | `ApiErrorResponse::ExternalConnectorError { code: "MISSING_FIELD", message: "...", ... }` |
| **Capture** fails | `Err(Status)` with `code: InvalidArgument` | `ConnectorError::ResponseHandlingFailed` | `ApiErrorResponse::InternalServerError` |
| **Direct connector** fails | `ConnectorError::MissingRequiredField` | Direct mapping | `ApiErrorResponse::MissingRequiredField { field_name: "..." }` |

**Current behavior:**
- Direct path preserves structured error types (best granularity)
- UCS authorize preserves error details as strings in `ExternalConnectorError`
- UCS other flows collapse to generic `InternalServerError`

### 2. HTTP Client Perspective (UCS HTTP API)

| Scenario | Authorize | Capture |
|----------|-----------|---------|
| Success | HTTP 200, `{"status": "success"}` | HTTP 200, `{"status": "success"}` |
| Connector error | HTTP 200, `{"error": "...", "error_code": "..."}` | HTTP 400/500, `{"error": {"message": "..."}}` |

**Current behavior:** HTTP status codes differ between flows for the same class of connector errors.

### 3. Error Granularity Comparison

| Path | Error Representation | Example |
|------|---------------------|---------|
| **Direct connector** | Structured `ApiErrorResponse` variants | `MissingRequiredField { field_name: "card.cvv" }` |
| **UCS Authorize** | `ExternalConnectorError` with string fields | `ExternalConnectorError { code: "MISSING_FIELD", message: "cvv is required" }` |
| **UCS Capture/Refund/etc** | Generic `InternalServerError` | `InternalServerError` (details lost) |

**What is `ExternalConnectorError`?**

`ExternalConnectorError` is a variant of the `ApiErrorResponse` enum (defined in `hyperswitch/main/crates/hyperswitch_domain_models/src/errors/api_error_response.rs:30`):

```rust
pub enum ApiErrorResponse {
    ExternalConnectorError {
        code: String,        // ← Free-form string from connector
        message: String,     // ← Free-form string from connector
        connector: String,
        status_code: u16,
        reason: Option<String>,
    },
    MissingRequiredField { field_name: String },  // ← Structured enum variant
    InternalServerError,
    // ... many other structured variants
}
```

## Solution Options

### Option 1: The Strict Two-Layer Model (Standard API Design)

This approach respects standard API protocol boundaries by separating "Integration/API failures" from "Business/Connector failures".

**a. Integration/Contract Errors (UCS-side):** Bad JSON, MissingRequiredField, AmountConversionFailed.

**gRPC Action:** Return `Err(tonic::Status::invalid_argument)`.

**HTTP Behavior:** The Axum layer automatically translates this to an HTTP 400 Bad Request (or 500 Internal Server Error).

**Why:** The request was malformed or the integration failed; the request never successfully made it through the business logic.

**b. Business/Connector Errors (Connector-side):** Card declined, invalid CVV, connector explicitly rejects the payload.

**gRPC Action:** Return `Ok(Response)` and populate the ErrorInfo field (e.g., connector_details.code = "DECLINED").

**HTTP Behavior:** HTTP 200 OK, with the error payload gracefully handled in the JSON body.

**Why:** The gRPC call succeeded and the server executed perfectly, but the payment itself failed at the downstream connector level.

**Changes:**

- Enhance Hyperswitch `UnifiedConnectorServiceError` handling
- Map `tonic::Code` to specific `ConnectorError` variants:
  - `InvalidArgument` → `MissingRequiredField`
  - `FailedPrecondition` → `FlowNotSupported`
  - `Unavailable` → Retryable error class
- Update `to_payment_failed_response` to preserve these mappings

**Pros:**
- Standard API design pattern
- Clear protocol-level vs business-level separation
- HTTP status codes follow REST conventions automatically

**Cons:**
- Loses rich error structure (limited to tonic::Code + message string) — But, this is actually solvable. Tonic supports the gRPC "Rich Error" model. We can serialize our ErrorInfo protobuf into a byte buffer and attach it directly to the tonic::Status using `Status::with_details()`. However, this requires extra work on the client side to deserialize the grpc-status-details-bin header.
- Harder to add payment-specific metadata

### Option 2: The Domain-First Model

This approach puts everything inside the ErrorInfo protobuf message and avoids gRPC protocol-level errors entirely (like current Authorize).

**Integration/Contract Errors (UCS-side):** Client forgot the "amount" field.

**gRPC Action:** Return `Ok(Response)` and populate ErrorInfo (e.g., unified_details.code = "MISSING_REQUIRED_FIELD", error_source = UCS).

**Business/Connector Errors (Connector-side):** Card declined.

**gRPC Action:** Return `Ok(Response)` and populate ErrorInfo (e.g., connector_details.code = "DECLINED", error_source = CONNECTOR).

**The HTTP Catch:** Because everything returns `Ok()`, the HTTP layer will default to returning 200 OK for everything. To maintain REST standards, the Axum handlers must explicitly inspect the error_source and manually override the HTTP status to 400 Bad Request if the source is UCS.

(Alternatively, in the long term, we could refactor the core logic to return a unified ApplicationErrorResponse, from which both HTTP and gRPC layers independently derive their respective formatting)

**Pros:**
- Rich error structure (all ErrorInfo fields available)
- Consistent pattern across all flows
- Natural fit for payment domain errors

**Cons:**
- **Metrics Skew:** Standard gRPC monitoring (Prometheus/Datadog) will log bad configurations and missing fields as "Successful" RPC calls, requiring us to build custom metrics tracking based on the payload.
- HTTP status code requires manual override logic in Axum handlers
- Diverges from standard gRPC conventions

## Error Source: Why Differentiate UCS vs Connector Errors?

An important consideration for any unified error handling approach is distinguishing between **two fundamentally different error sources:**

### Error Source Types

| Error Source | Cause | Example | Actionable By |
|--------------|-------|---------|---------------|
| **UCS-side** | Integration or infrastructure issues | HS didn't send required field, bad connector config, proto parsing failure | HS or UCS team |
| **Connector-side** | Normal business outcomes from external connector | Card declined, insufficient funds, invalid CVV | Merchant (retry/different card) |

### Why This Matters

**1. Monitoring & Alerting**
- **UCS errors:** Should trigger alerts (integration bugs, need fixing)
- **Connector errors:** Normal business behavior (don't alert, just track rate)

**2. SLO Definition**
- UCS availability SLO should **exclude** connector errors
- Connector declining cards is not a UCS failure

**3. Debugging & Analytics**
- Quickly identify if issue is HS<>UCS integration bug vs normal payment failure
- Track connector-specific error rates for analytics

### Implementation Approach

Add `error_source` field to error details:

```protobuf
message UnifiedErrorDetails {
  optional string code = 1;
  optional string message = 2;
  optional string description = 3;
  optional string user_guidance_message = 4;
  optional ErrorSource source = 5;  // NEW field
}

enum ErrorSource {
  ERROR_SOURCE_UNSPECIFIED = 0;
  ERROR_SOURCE_CONNECTOR = 1;  // Error from external connector API
  ERROR_SOURCE_UCS = 2;         // Error from UCS infra/integration
}
```

**Usage:**
- Set `ERROR_SOURCE_CONNECTOR` for payment failures (decline, invalid data from connector)
- Set `ERROR_SOURCE_UCS` for integration issues (missing required field from HS, bad config, timeout calling connector)

## Architecture Overview

### 1. **Hyperswitch Direct Connector Path**
- Hyperswitch calls connector APIs directly via `api_client.rs`
- Connector responses mapped to `RouterData.response: Result<PaymentsResponseData, ErrorResponse>`
- `ConnectorError` variants preserved and mapped via `to_payment_failed_response`

### 2. **Hyperswitch → UCS gRPC Path**
- Hyperswitch uses gRPC client to call UCS (`external_services/.../unified_connector_service.rs`)
- UCS executes connector logic and returns response
- Response handling differs between authorize and other flows

### 3. **UCS gRPC Service**
- **Authorize:** Custom implementation in `grpc-server/.../server/payments.rs`
- **Other flows:** Macro-generated in `grpc-server/.../utils.rs` (`implement_connector_operation!`)

### 4. **UCS HTTP (Axum)**
- REST interface wrapping same gRPC service
- Handlers convert JSON → `tonic::Request` → call gRPC service → JSON response
- `tonic::Status` mapped to HTTP status codes in `http/error.rs`

## Current State: Error Handling Flows

### Flow 1: Authorize (UCS gRPC)

**Path:** `connector-service/crates/grpc-server/grpc-server/src/server/payments.rs` (~668–1100)

```rust
// Actual flow
async fn process_authorization_internal(...) -> Result<PaymentServiceAuthorizeResponse, PaymentAuthorizationError> {
    // Pre-processing (bad config, bad payload) → PaymentAuthorizationError
    let connectors = get_resolved_connectors(...).map_err(|e| PaymentAuthorizationError::new(...))?;
    let payment_flow_data = PaymentFlowData::foreign_try_from(...).map_err(|e| PaymentAuthorizationError::new(...))?;

    let response = execute_connector_processing_step(...).await;

    let authorize_response = match response {
        Ok(success_response) => {
            // Business errors (4xx/5xx) already in router_data.response = Err(ErrorResponse)
            generate_payment_authorize_response(success_response)?
        }
        Err(error_report) => {
            // Integration errors (timeout, etc) → manually construct ErrorResponse
            let app_err: ApplicationErrorResponse = error_report.current_context().switch();
            let error_router_data = RouterDataV2 {
                response: Err(ErrorResponse {
                    code: app_err.sub_code,
                    message: app_err.error_message,
                    ...
                })
            };
            generate_payment_authorize_response(error_router_data)?  // Populates ErrorInfo
        }
    };
    Ok(authorize_response)
}

pub async fn authorize(...) -> Result<Response<PaymentServiceAuthorizeResponse>, Status> {
    match process_authorization_internal(...).await {
        Ok(response) => Ok(Response::new(response)),  // ← Always Ok, errors in ErrorInfo
        Err(PaymentAuthorizationError) => {
            // Only pre-processing or response generation failures
            let error_response = PaymentServiceAuthorizeResponse::from(error);
            Ok(Response::new(error_response))
        }
    }
}
```

**Key behavior:**
- **Connector business errors (4xx/5xx):** Created as `ErrorResponse` by connector implementation → `generate_payment_authorize_response` populates ErrorInfo
- **Integration errors (timeout, bad request):** Converted to `ErrorResponse` in error path → `generate_payment_authorize_response` populates ErrorInfo
- **Pre-processing failures (bad config, bad payload):** → `PaymentAuthorizationError` → ErrorInfo
- **All errors** end up in ErrorInfo field
- **Returns `Ok(tonic::Response)`** for all cases

**What happens to these errors in Hyperswitch:**

**Path:** `hyperswitch/main/crates/router/src/core/errors/utils.rs` (~199–306)

When Hyperswitch has `RouterData.response = Err(ErrorResponse)` (from UCS authorize returning `Ok` with error fields):
- The `ErrorResponse` contains `code`, `message`, `reason` as **strings**
- These typically get wrapped into `ApiErrorResponse::ExternalConnectorError` with the error details preserved as string fields
- **Not** converted to structured variants like `MissingRequiredField` (those only occur for Direct path or HS pre-validation)

When UCS returns `Err(tonic::Status)` (capture/refund/etc):
- Wrapped as `ConnectorError::ResponseHandlingFailed`
- Maps to `ApiErrorResponse::InternalServerError` (generic 500)
- The original `tonic::Code` (e.g., `InvalidArgument`) is not currently mapped to specific error types

## Characteristics of Current State

### Actual Error Flow (Correcting Initial Assumptions)

Initial hypothesis suggested that UCS authorize errors would map to structured `ApiErrorResponse::MissingRequiredField`. **Investigation revealed this is not accurate:**

**Actual flow for UCS authorize errors:**

1. UCS returns `Ok(Response)` with `error_code`, `error_message` fields
2. Hyperswitch `foreign_try_from` creates `ErrorResponse { code: String, message: String, ... }`
3. This gets wrapped as `ApiErrorResponse::ExternalConnectorError` with the code/message as **strings**
4. **Not** converted to structured variants like `MissingRequiredField`

**Comparison:**

| Source | Error Type | Final API Response | Structured? |
|--------|-----------|-------------------|-------------|
| **HS Direct** | `ConnectorError::MissingRequiredField` | `ApiErrorResponse::MissingRequiredField { field_name }` | ✓ Yes - enum variant |
| **UCS Authorize** | `ErrorResponse { code: "MISSING_FIELD" }` | `ApiErrorResponse::ExternalConnectorError { code: "MISSING_FIELD" }` | ✗ No - string field |
| **UCS Capture** | `tonic::Status(InvalidArgument)` | `ApiErrorResponse::InternalServerError` | ✗ No - generic error |

**Implication:** All three paths have different error granularity:
- Direct: Best (structured types)
- UCS Authorize: Medium (details preserved as strings)
- UCS Capture/etc: Worst (details lost)
