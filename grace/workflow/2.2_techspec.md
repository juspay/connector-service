# Tech Spec Agent

You generate the tech spec for **{CONNECTOR}**. Two execution paths are supported:

- **Path A — Grace CLI** (`grace techspec`) when `grace/.env` is configured with real API keys.
- **Path B — Claude-Native** when `grace/.env` is missing or contains placeholder values. Path B follows the workflow at `.skills/generate-tech-spec/references/techspec-generation-native.md` and uses Claude tools (WebFetch + in-place edits) to produce the same canonical spec.

The environment check in Phase 1c selects the path automatically. This must complete before any code work begins.

---

## Inputs

| Parameter                          | Description                                       | Example                      |
| ---------------------------------- | ------------------------------------------------- | ---------------------------- |
| `{CONNECTOR}` / `{Connector_Name}` | Connector name (exact casing from connector list) | `Adyen`                      |
| `{FLOW}`                           | Payment flow being implemented                    | `BankDebit`, `MIT`, `Wallet` |

Use `{Connector_Name}` (exact casing) for `grace techspec` commands. Use `{connector}` (lowercase) for file names.

---

## Phase 1a: Extract URLs for the connector

The Links Agent has already gathered documentation URLs and written them to `data/integration-source-links.json`. Extract the URLs for this connector:

```bash
# From connector-service root:
cat data/integration-source-links.json | jq -r '."{Connector_Name}" // ."{connector}" // empty | .[]' 2>/dev/null
```

If `data/integration-source-links.json` does not exist or has no entry for this connector, return FAILED with reason "No documentation links found for {CONNECTOR}. Run the Links Agent (2.1_links.md) first."

---

## Phase 1b: Create the URL file

Create a file `{connector_name}.txt` (lowercase) containing the URLs, one per line:

```bash
# From connector-service root:
cat data/integration-source-links.json | jq -r '."{Connector_Name}" // ."{connector}" // empty | .[]' > {connector_name}.txt
```

Verify the file is non-empty:

```bash
wc -l {connector_name}.txt
```

If the file is empty (0 lines), return FAILED with reason "No URLs extracted for {CONNECTOR}."

---

## Phase 1c: Environment Check (select Path A or Path B)

Before generating the spec, detect whether the grace CLI is configured. The check verifies that `grace/.env` exists **and** contains real (non-placeholder) values for `AI_API_KEY` and `FIRECRAWL_API_KEY`:

```bash
# From connector-service root:
if [ -f grace/.env ] \
   && grep -q '^AI_API_KEY=' grace/.env \
   && ! grep -q '^AI_API_KEY=your_' grace/.env \
   && grep -q '^FIRECRAWL_API_KEY=' grace/.env \
   && ! grep -q '^FIRECRAWL_API_KEY=your_' grace/.env; then
  echo "GRACE_ENV_READY"
else
  echo "GRACE_ENV_MISSING"
fi
```

- `GRACE_ENV_READY` → continue to **Phase 2a: Path A — Grace CLI**.
- `GRACE_ENV_MISSING` → continue to **Phase 2b: Path B — Claude-Native**.

This single check catches a missing `.env`, an empty `.env`, or an `.env` whose values are still the `your_…` placeholders copied from `.env.example`.

---

## Phase 2a: Path A — Grace CLI (when GRACE_ENV_READY)

```bash
# IMPORTANT: Switch to grace/ directory and activate virtualenv
# workdir: connector-service/grace
source .venv/bin/activate
cat ../{connector_name}.txt | grace techspec {Connector_Name} -e
```

**Wait for the `grace techspec` command to complete before proceeding.** This can take approximately 20 minutes. Do NOT interrupt or proceed until the techspec is fully generated.

**Critical:**

- The working directory MUST be `grace/` when running this command.
- The virtual environment MUST be activated first.
- The `-e` flag is required (enables Claude Agent SDK enhancement + field-dependency analysis).

Path A writes the spec under `grace/rulesbook/codegen/references/` (legacy path: `specs/{ConnectorName}.md`).

Skip Phase 2b — proceed to Phase 3 (Verify).

---

## Phase 2b: Path B — Claude-Native (when GRACE_ENV_MISSING)

The grace CLI is not available in this environment. Generate the spec using Claude tools instead by following the canonical Claude-native workflow at:

```
.skills/generate-tech-spec/references/techspec-generation-native.md
```

That workflow file is self-contained. Read it and execute its steps with these inputs:

- `CONNECTOR` / `CONNECTOR_NAME` = `{Connector_Name}` (exact casing)
- `FLOW` = `{FLOW}`
- URL list: from `data/integration-source-links.json` (already extracted in Phase 1a) or from `{connector_name}.txt` (already created in Phase 1b)

**Expected outputs from Path B:**

- Scraped sources at `grace/rulesbook/codegen/references/{connector}/source_{N}.md` (one per URL)
- The generated tech spec at `grace/rulesbook/codegen/references/specs/{ConnectorName}.md` (same path Path A writes to)
- Sections 10/11/12 (`## API Call Sequences`, `## Field Dependency Analysis`, `## UNDECIDED Fields`) appended to the spec by the field-analysis step inside that workflow

**Limitations vs Path A:**

- Web scraping uses WebFetch, which may not render JavaScript-heavy documentation pages as well as Firecrawl.
- Sources are processed sequentially (one URL at a time) rather than in parallel.
- No mock-server generation (the grace CLI `-m` flag has no equivalent here).

Path B is functionally equivalent for downstream consumers: the spec it produces conforms to the canonical 12-section schema in `.skills/generate-tech-spec/references/techspec-generation-native.md` and lands where consumer skills (`new-connector`, `add-connector-flow`, `add-payment-method`) look for it.

Proceed to Phase 3 (Verify) once Path B completes.

---

## Phase 3: Verify tech spec output

After either path finishes, verify the spec was created:

```bash
# From connector-service root:
find grace/rulesbook/codegen/references -iname "*{connector}*" | head -20
```

If no spec was generated, return FAILED. Store the resolved path as `{TECHSPEC_PATH}` for Phase 4.

---

## Phase 4: Publish Tech Spec Issue to `juspay/grace` (HARD GATE)

**Purpose**: Every generated tech spec MUST be published as a GitHub issue in `juspay/grace` so that the corresponding implementation PR (created later by the PR Agent) can reference it for traceability.

**HARD GATE — NON-NEGOTIABLE**: This phase is a HARD GATE. The agent does NOT report SUCCESS unless an issue exists in `juspay/grace` for this connector+flow AND a non-empty `ISSUE_URL` / `ISSUE_NUMBER` are returned. There are no exceptions. If issue creation cannot succeed (auth error, repo unreachable, network failure, body too large after truncation, etc.), the agent MUST return STATUS=FAILED with REASON describing the failure. Downstream agents (Codegen, PR) MUST NOT proceed without this cross-reference.

**HARD GUARDRAIL — TARGET REPOSITORY**: ALL `gh` commands in this phase MUST use `--repo juspay/grace`. Do NOT target any other repo.

**Pre-flight check (run once before 4a)**:

```bash
# Verify gh is authenticated and has access to juspay/grace before doing anything else.
gh auth status >/dev/null 2>&1 || { echo "ERROR: gh not authenticated"; exit 1; }
gh repo view juspay/grace >/dev/null 2>&1 || { echo "ERROR: cannot access juspay/grace"; exit 1; }
```

If either check fails, return STATUS=FAILED immediately with REASON=`"Tech spec issue gate blocked: <reason>"`. Do NOT skip this phase, do NOT proceed to Output with empty issue fields.

### 4a: Read the sources used

```bash
# From connector-service root:
cat data/integration-source-links.json | jq -r '."{Connector_Name}" // ."{connector}" // empty | .[]'
```

Store the URL list. If empty, proceed anyway with a `Sources: (none recorded)` note — the issue is still useful.

### 4b: Read the generated tech spec

```bash
# From connector-service root:
wc -c {TECHSPEC_PATH}    # check size
cat {TECHSPEC_PATH}      # read the full spec
```

Capture the byte count as `{SPEC_BYTES}`.

### 4c: Idempotency check — does an open issue already exist?

```bash
gh issue list \
  --repo juspay/grace \
  --state open \
  --label "tech-spec" \
  --search "[Tech Spec] {Connector_Name} — {FLOW} in:title" \
  --json number,url,title \
  --jq '.[0]'
```

If a matching issue is returned (same connector + flow in the title), reuse it: set `{ISSUE_URL}` and `{ISSUE_NUMBER}` to the existing values and skip to the Output. Do NOT create a duplicate.

### 4d: Ensure labels exist (idempotent)

```bash
gh label create "tech-spec" --repo juspay/grace --description "Generated tech spec for a payment connector flow" --color "0e8a16" 2>/dev/null || true
gh label create "GRACE" --repo juspay/grace --description "Auto-generated by GRACE pipeline" --color "7057ff" 2>/dev/null || true
```

### 4e: Compose the issue body

Use this **exact structured template**. Substitute the variables; do NOT add other sections.

```markdown
## Tech Spec: {Connector_Name} — {FLOW}

| Field | Value |
| --- | --- |
| Connector | `{Connector_Name}` |
| Flow | `{FLOW}` |
| Generation Path | `A` (grace CLI) _or_ `B` (Claude-native) |
| Spec File | `{TECHSPEC_PATH}` (in `juspay/hyperswitch-prism`) |
| Generated | <ISO-8601 UTC date, e.g. `2026-05-18`> |
| Generated By | GRACE Tech Spec Agent (`grace/workflow/2.2_techspec.md`) |

---

## Sources

Official documentation URLs used to derive this spec:

1. <url 1>
2. <url 2>
...

_(If no sources were recorded, write `_(none recorded)_`.)_

---

## Tech Spec

<full content of {TECHSPEC_PATH}, fenced as ```markdown ... ``` ONLY if the spec is itself markdown; otherwise paste raw>

---

> Auto-generated by **GRACE**. The implementation PR opened by the PR Agent (`2.4_pr.md`) in `juspay/hyperswitch-prism` will reference this issue as `juspay/grace#<this-number>`.
```

**Size handling**: If `{SPEC_BYTES}` exceeds **60000**, the issue body would be rejected. In that case, replace the `## Tech Spec` section content with:

```markdown
> The full tech spec exceeds GitHub's issue body limit. See the file at
> [`{TECHSPEC_PATH}`](https://github.com/juspay/hyperswitch-prism/blob/main/{TECHSPEC_PATH})
> once the implementation PR is merged. Below is a truncated excerpt:

<first ~50000 characters of the spec, fenced as markdown>
```

### 4f: Create the issue

```bash
gh issue create \
  --repo juspay/grace \
  --title "[Tech Spec] {Connector_Name} — {FLOW}" \
  --label "tech-spec" \
  --label "GRACE" \
  --body "$(cat <<'EOF'
<body composed in 4e>
EOF
)"
```

Capture the returned URL. Parse the issue number from the trailing `/issues/<N>` segment.

Store as `{ISSUE_URL}` and `{ISSUE_NUMBER}`.

### 4g: Failure handling (HARD GATE)

If `gh issue create` fails (auth error, repo unreachable, label creation race, body too large even after truncation, rate limit, etc.):

- Return STATUS=`FAILED`.
- Set REASON=`"Tech spec issue gate failed: <verbatim gh error output>"`.
- Leave `ISSUE_URL` and `ISSUE_NUMBER` empty in the Output.

**Retry policy**: Retry `gh issue create` exactly ONCE after a 5-second wait if the error is transient (network timeout, 5xx, rate limit with a Retry-After). Do NOT retry on auth errors, permission denied, validation errors (4xx other than 429), or body-too-large — those are deterministic and will produce the same error.

**No bypass**: Do NOT return SUCCESS with empty issue fields. Do NOT skip Phase 4. Do NOT continue silently. The PR Agent (`2.4_pr.md`) and Connector Agent (`2_connector.md`) explicitly forbid proceeding without a valid `ISSUE_NUMBER`, so returning SUCCESS without one would only push the failure downstream where it produces a worse error.

### 4h: Post-creation verification

After `gh issue create` returns a URL, verify the issue is actually retrievable (guards against URL parsing bugs and transient writes that didn't persist):

```bash
gh issue view {ISSUE_NUMBER} --repo juspay/grace --json number,url,title --jq '.number'
```

If this command fails or returns a number that does not match `{ISSUE_NUMBER}`, treat it as a creation failure and apply 4g.

**Contract for SUCCESS**: STATUS=SUCCESS REQUIRES all four to be non-empty: `TECHSPEC_PATH`, `ISSUE_URL`, `ISSUE_NUMBER` (non-zero integer), and a tracking issue that exists in `juspay/grace`. Any other combination MUST be reported as STATUS=FAILED.

## Rules

1. **No code generation** — this agent only produces the tech spec and its tracking issue, nothing else.
2. **Verify before reporting success** — always confirm the file exists on disk.
3. **Clean up on failure** — if the chosen path fails, capture the error output and include it in the `REASON` field. Do NOT silently fall over from Path A to Path B mid-run; the Phase 1c check selects the path, and the chosen path either succeeds or fails.
4. **Idempotent — spec** — if a tech spec already exists for this connector, report SUCCESS with the existing path (do not regenerate).
5. **Idempotent — issue** — Phase 4c MUST search for an existing open `tech-spec` issue with the same title and reuse it. Never open duplicate issues.
6. **Issue creation IS a hard gate** — if Phase 4 cannot produce a valid `ISSUE_URL` + `ISSUE_NUMBER` pointing at a real issue in `juspay/grace`, return STATUS=FAILED. SUCCESS without an issue is FORBIDDEN. Downstream agents will refuse to proceed without these fields; bypassing this gate produces a worse failure downstream.
7. **URL source** — URLs come from `data/integration-source-links.json`, written by the Links Agent. Do NOT expect or require an integration details file.
8. **Canonical schema** — regardless of path, the produced spec must match `.skills/generate-tech-spec/references/techspec-generation-native.md` (12 sections + appended Sections 10/11/12).
9. **Target repo for issues is `juspay/grace` — ALWAYS** — every `gh` command in Phase 4 MUST include `--repo juspay/grace`. The implementation PR (different repo, `juspay/hyperswitch-prism`) is opened later by the PR Agent.
