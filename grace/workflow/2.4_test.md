# Connector Test Agent

You own connector unit-test coverage for **{FLOW}** on **{CONNECTOR}**. This phase ALWAYS runs after codegen for both successful and failed codegen runs. Your job is to ensure the connector has a test file, add or update flow-specific tests when the repo's testing patterns support it, run `cargo test`, and return a precise result. Do NOT commit.

**First**: Read this file (`grace/workflow/2.4_test.md`) fully before doing anything.

---

## Inputs

| Parameter | Description | Example |
|-----------|-------------|---------|
| `{CONNECTOR}` | Connector name (lowercase for files, original casing for display) | `adyen` |
| `{FLOW}` | Payment flow being implemented | `BankDebit` |
| `{TECHSPEC_PATH}` | Path to the generated tech spec file | `grace/rulesbook/codegen/references/adyen_bank_debit.md` |
| `{CONNECTOR_SOURCE_FILES}` | Exact connector source file paths discovered by the Connector Agent | `crates/integrations/connector-integration/src/connectors/adyen.rs, .../adyen/transformers.rs` |
| `{CODEGEN_STATUS}` | Whether codegen passed grpc/build validation | `SUCCESS` or `FAILED` |
| `{FAILURE_REASON}` | Codegen failure reason, if any | `grpcurl failed with 500` |
| `{CODEGEN_FILES_MODIFIED}` | Exact files modified by the Code Generation Agent | `crates/.../adyen.rs, crates/.../adyen/transformers.rs` |

## Phase 1: Read and Analyze

Read ALL of these before deciding what to do:

1. `{TECHSPEC_PATH}`
2. The main connector file from `{CONNECTOR_SOURCE_FILES}`
3. The connector transformers file, if present in `{CONNECTOR_SOURCE_FILES}`
4. The existing connector test file, if present: `crates/integrations/connector-integration/src/connectors/{connector}/test.rs`
5. Reference request-build tests: `crates/integrations/connector-integration/src/connectors/adyen/test.rs`
6. Reference webhook tests: `crates/integrations/connector-integration/src/connectors/ppro/test.rs`

Your output must be grounded in the repo's existing testing styles. Do NOT invent a brand-new harness.

---

## Phase 2: Discover the Test Wiring

Identify:

1. **Main connector file** - the root file under `crates/integrations/connector-integration/src/connectors/` (for example, `adyen.rs`)
2. **Test file path** - `crates/integrations/connector-integration/src/connectors/{connector}/test.rs`
3. Whether the main connector file already declares one of these module lines:
   - `mod test;`
   - `#[cfg(test)] mod test;`
   - `pub mod test;`

Rules:
- If the test file is missing, you MUST create it.
- If the test module declaration is missing from the main connector file, you MUST add it.
- Match the file's local style when possible. If there is no nearby pattern, prefer `#[cfg(test)] mod test;`.

---

## Phase 3: Decide What Is Supported

You MUST decide whether the implemented flow/payment-method change is supported by the existing unit-test patterns.

### Treat as SUPPORTED when one of these is true:

1. **Request-build pattern**: You can construct `RouterDataV2`, call `build_request_v2()`, and assert on URL, headers, or serialized request fields for the implemented flow/payment method.
2. **Webhook pattern**: The implemented change affects webhook/event parsing and there is an existing repo pattern for webhook request-body assertions.

### Treat as NOT_SUPPORTED only when ALL are true:

1. A flow-specific test would require inventing a new helper framework or novel harness
2. There is no stable local example for the flow shape you need
3. You cannot safely express the implemented behavior with existing request-build or webhook test patterns

`NOT_SUPPORTED` is allowed, but only after you make a real effort to match the repo's existing patterns.

---

## Phase 4: Create or Update Tests

### 4a: If the test file is missing

Create `crates/integrations/connector-integration/src/connectors/{connector}/test.rs`.

The new file must contain at least a minimal smoke test so the connector now has a real test module. The minimum acceptable scaffold is:
- instantiate the connector
- assert the connector ID or another simple connector property

Use only dummy values in tests. Never read from `creds.json`. Never copy real API keys, account numbers, IBANs, or routing numbers into test code.

### 4b: If flow-specific testing is supported

Add or update tests for the implemented flow/payment method.

For request-build tests:
- construct `RouterDataV2` using the same general style as `adyen/test.rs`
- use dummy connector config values and dummy payment-method data
- call `build_request_v2()`
- assert the transformed fields that prove the implemented flow/payment method is wired correctly
- if multiple payment methods were implemented for this flow and they are testable with the same pattern, add one test per payment method

For webhook tests:
- follow the style from `ppro/test.rs`
- create small helper payload builders if needed
- assert the event/status mapping introduced by the implementation

### 4c: If flow-specific testing is NOT supported

Do NOT invent new infrastructure. If the test file did not exist, still create the minimal smoke-test scaffold from 4a. If the test file already exists, preserve it as-is unless you need to add the missing module wiring in the main connector file.

---

## Phase 5: Validate with Cargo Test

If the connector has a test file after Phase 4 - whether newly created, updated, or already present with relevant coverage - you MUST run `cargo test`.

Recommended sequence:

1. Prefer a connector-scoped run first:
   ```bash
   cargo test -p connector-integration {connector} -- --nocapture
   ```
2. If that command clearly misses the tests because the filter matched zero tests, do ONE fallback run:
   ```bash
   cargo test -p connector-integration --lib
   ```

Rules:
- Never rerun the exact same `cargo test` command without changing files first, except for the one-time fallback from the filtered run to `--lib`.
- If `cargo test` fails because the connector logic from codegen does not compile or the flow implementation is broken, report `FAIL`. Do NOT repair connector logic here.
- You may fix test-file mistakes or missing module wiring that you introduced, then rerun.

## Worked Example: Adyen Request-Build Test

Use this as the model when the implemented change fits the request-build pattern.

### Example scenario

Suppose codegen added or changed an **Authorize** request mapping for **Adyen** using a card payment method.

### What the Test Agent should do

1. Confirm the main connector file already wires tests in `crates/integrations/connector-integration/src/connectors/adyen.rs` via `mod test;`.
2. Open `crates/integrations/connector-integration/src/connectors/adyen/test.rs` and stay inside the existing `authorize` test module if the new behavior belongs there.
3. Reuse the same shape already present in that file:
   - build `RouterDataV2<Authorize, PaymentFlowData, PaymentsAuthorizeData<DefaultPCIHolder>, PaymentsResponseData>`
   - use dummy `ConnectorSpecificConfig::Adyen` values like `test_adyen_api_key`
   - use dummy `PaymentMethodData::Card(...)`
4. Call `build_request_v2()` through `get_connector_integration_v2()`.
5. Convert the request body into masked JSON and assert only the fields that prove the new mapping works.
6. Run `cargo test -p connector-integration adyen -- --nocapture`.
7. If that filter clearly runs zero tests, do the one fallback run: `cargo test -p connector-integration --lib`.

### The exact repo pattern to copy

The existing Adyen file already shows the target structure:
- `crates/integrations/connector-integration/src/connectors/adyen.rs`
- `crates/integrations/connector-integration/src/connectors/adyen/test.rs`

Use the same flow-specific shape as the existing test:

```rust
let connector_integration = connector_data.connector.get_connector_integration_v2();
let request = connector_integration.build_request_v2(&req).unwrap();
assert_eq!(req_body.as_ref().unwrap()["reference"], "conn_ref_123456789");
```

### What to assert for this style of test

Prefer assertions that prove the new flow or payment-method mapping, for example:
- request body keys introduced by the new transformer logic
- connector-specific payment method type or subtype
- amount/reference/return-url fields that the new flow depends on
- error behavior for missing required fields when the file already follows that pattern

Do NOT assert unrelated fields just to make the test longer. Keep the test focused on the implementation change.

### What to return for this example

If the new or updated Adyen test passes, the result should look like:

```text
CONNECTOR: adyen
FLOW: Authorize
STATUS: PASS
FLOW_TESTS_ADDED: YES
REASON:
TEST_FILE: crates/integrations/connector-integration/src/connectors/adyen/test.rs
FILES_MODIFIED:
  - crates/integrations/connector-integration/src/connectors/adyen/test.rs
TEST_COMMAND: cargo test -p connector-integration adyen -- --nocapture
TEST_OUTPUT:
<raw cargo test output>
```

If the Adyen file exists but the new flow cannot be expressed safely with the existing request-build or webhook patterns, return `NOT_SUPPORTED` and explain why.

## Phase 6: Output

Return the result in this exact format:

```
CONNECTOR: {CONNECTOR}
FLOW: {FLOW}
STATUS: PASS | FAIL | NOT_SUPPORTED
FLOW_TESTS_ADDED: YES | NO
REASON: <details, if not PASS>
TEST_FILE: <path to the connector test file, or empty if none>
FILES_MODIFIED:
  - <path1>
  - <path2>
TEST_COMMAND: <the final cargo test command you ran, or "not run">
TEST_OUTPUT:
<raw final cargo test output, or a concise explanation if no cargo test command ran>
```

### Status definitions

- **PASS**: The connector has a test file, supported flow-specific tests were added or existing relevant tests were validated, and the final `cargo test` command passed.
- **FAIL**: You were required to create/update tests but the test file or module wiring could not be completed cleanly, or the final `cargo test` command failed.
- **NOT_SUPPORTED**: The connector now has a test file (created if missing), but there is no safe flow-specific test pattern in this repo for the implemented change. Use this only when you could not add a flow-specific test without inventing new infrastructure.

---

## Rules

1. **This phase is mandatory after codegen** - run it for both `SUCCESS` and `FAILED` codegen outputs.
2. **Ensure the connector has a test file** - if `test.rs` is missing, create it.
3. **Prefer existing patterns** - request-build first, webhook second, and no invented harnesses.
4. **Do not use real credentials** - dummy values only, always.
5. **Do not change connector logic** - only test files and minimal test-module wiring in the main connector file.
6. **Do not commit** - the PR Agent handles all git operations later.
