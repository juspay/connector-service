# Connector Test Agent

You own scenario-driven test coverage for **{CONNECTOR}** using the `ucs-connector-tests` harness. This phase covers ALL implemented flows and payment methods, not just a single `{FLOW}`. It ALWAYS runs after each codegen attempt. Your job is to ensure the connector has test specs for every implemented flow that has a global suite, add or update payment-method scenarios when needed, run the scenario harness, and return precise repair feedback. Do NOT commit.

**First**: Read this file (`grace/workflow/2.4_test.md`) fully before doing anything.

**Important**: Any result other than `PASS` feeds back into the Connector Agent's retry loop. Be strict, specific, and actionable.

## Framework Overview

Tests live in `crates/internal/ucs-connector-tests/`. This is a scenario-driven harness — you do NOT write Rust test code. You edit JSON files only.

### Key paths

| What | Path |
|------|------|
| Global suite scenarios | `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/scenario.json` |
| Global suite dependency spec | `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/suite_spec.json` |
| Connector support matrix | `crates/internal/ucs-connector-tests/src/connector_specs/<connector>/specs.json` |
| Connector-specific overrides | `crates/internal/ucs-connector-tests/src/connector_specs/<connector>/override.json` |

### Existing global suites

|------------|------|

| `capture` | Capture (full, partial, with merchant order ID) |

|-----------------------------------|------------|

## Phase 0: Discovery (standalone mode or missing inputs)

**When to run Phase 0**: Run this phase if any of the following are true:
- `{PR_NUMBER}` is provided (standalone mode)
- `{IMPLEMENTED_FLOWS}` is missing or set to `unknown`
- `{IMPLEMENTED_PAYMENT_METHODS}` is missing or set to `unknown`

If all pipeline inputs are present and `{IMPLEMENTED_FLOWS}` is a real comma-separated list (not `unknown`), skip Phase 0 and go directly to Phase 1.

### 0a: If `{PR_NUMBER}` is provided — get PR context

```bash
gh api repos/{REPO}/pulls/{PR_NUMBER}/files --jq '.[].filename'
```

From the file list:
- Infer `{CONNECTOR}` from connector file paths (e.g. `connectors/razorpayv2.rs` → `razorpayv2`)
- Identify `{CONNECTOR_SOURCE_FILES}` (files under `crates/integrations/connector-integration/src/connectors/`)
- Identify which test spec files already exist in the PR

If not already on the PR branch, check it out:
```bash
git fetch origin <branch_name> && git checkout <branch_name>
```

Set defaults for missing pipeline inputs:
- `{CODEGEN_STATUS}` = `SUCCESS`
- `{CODEGEN_GRPCURL_RESULT}` = `NOT_RUN`
- `{ATTEMPT_NUMBER}` = `1`

### 0b: Discover implemented flows

Read the connector's main file (`crates/integrations/connector-integration/src/connectors/{connector}.rs`):

1. **If `create_all_prerequisites!` macro exists**: extract every `flow: <FlowName>` entry from the `api: [...]` block.
2. **If no macro (manual impl blocks)**: find all `impl ConnectorIntegrationV2<FlowName, ...>` blocks. Include ALL flows — both fully implemented and stubs. The test harness uses `specs.json` to declare which suites to run, and the suite runner will naturally fail for flows that are not implemented yet. This is preferable to undercounting, because schema drift checks catch invalid suite declarations and the suite runner provides actionable failure output.
3. Store as `{IMPLEMENTED_FLOWS}` (comma-separated).

### 0c: Discover implemented payment methods

Read the connector's transformers file (`crates/integrations/connector-integration/src/connectors/{connector}/transformers.rs`):

1. Find the `PaymentMethodData` match in the Authorize request `TryFrom` impl.
2. Extract every `PaymentMethodData::<Variant>` that maps to `Self::try_from(...)`, `Ok(...)`, or any non-error arm.
3. Ignore variants that fall through to `Err(ConnectorError::NotImplemented(...))`.
4. If the match uses a wildcard `_ =>` catch-all, only list the explicitly named handled variants.
5. If no `PaymentMethodData` match exists (payment-method-agnostic connector), set to `unknown`.
6. Report top-level variant names only (e.g. `Card`, not `Card::Credit`).
7. Store as `{IMPLEMENTED_PAYMENT_METHODS}` (comma-separated).

## Phase 2: Build the Action Plan

You do NOT operate on a single flow. You iterate over ALL flows in `{IMPLEMENTED_FLOWS}` and build a combined action plan.

### Step 1: Map each flow to a suite

For each flow in `{IMPLEMENTED_FLOWS}`, use the flow-to-suite mapping table above. Collect all suite names that have a matching global suite directory.

**Example**: If `{IMPLEMENTED_FLOWS}` is `Authorize,PSync,Capture,Refund,RSync`, the mapped suites are `authorize,get,capture,refund,refund_sync`.

Flows that have no matching global suite (e.g. `CreateOrder`) are skipped for Case A/B and noted in the output.

### Step 2: Determine which suites are already in `specs.json`

Read `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/specs.json` (if it exists). Compare the mapped suites from Step 1 against `supported_suites`. Any suite not yet listed needs to be added.

### Step 3: Check payment methods for the authorize suite

**Prerequisite**: Case B (adding new global scenarios) requires `{CODEGEN_GRPCURL_RESULT}` = `PASS`. If grpcurl did not run or failed, skip Step 3 entirely. Do NOT create new global scenarios without grpcurl validation — you have no way to verify the payment method actually works, and a bad scenario will break schema checks for every connector.

If `{CODEGEN_GRPCURL_RESULT}` is `PASS`, `authorize` is one of the mapped suites, and `{IMPLEMENTED_PAYMENT_METHODS}` is not `unknown`:
- Read `crates/internal/ucs-connector-tests/src/global_suites/authorize_suite/scenario.json`
- Check which payment methods already have scenarios (currently: `Card` only)
- For each payment method in `{IMPLEMENTED_PAYMENT_METHODS}` that is NOT `Card`:
  - That payment method needs a new scenario in `authorize_suite/scenario.json` → mark as Case B

If `{CODEGEN_GRPCURL_RESULT}` is `FAIL` or `NOT_RUN`, skip this step. Only Case A (enable existing suites in `specs.json`) is safe without grpcurl validation.

### Step 4: Classify the action

| Situation | Case |
|-----------|------|
| Suite exists in global_suites AND not in connector's specs.json | Case A — add suite to specs.json |
| Authorize suite exists, a new payment method needs a scenario, AND `{CODEGEN_GRPCURL_RESULT}` is `PASS` | Case B — add global scenario + add/update override |
| No global suite exists for a flow | Case C — create new suite (rare) or skip |
| Change is purely internal (webhook, error mapping) | Case D — NOT_SUPPORTED |

Combine all actions. A single test-agent run may apply Case A for 5 suites AND Case B for one payment method at the same time.

### Step 5: Derive connector-specific overrides from grpcurl output

If `{CODEGEN_GRPCURL_OUTPUT}` is available and non-empty:

1. Parse the grpcurl response JSON and extract the `status` field value (e.g. `"authorized"`, `"PENDING"`, `"AUTHENTICATION_PENDING"`, `"CHARGED"`)
2. For the `authorize` suite, read the global `authorize_suite/scenario.json` and find the matching scenario's `assert.status.one_of` list
3. If the observed status is NOT already in that `one_of` list, add a connector override in `override.json` that extends `one_of` to include the observed status
4. If the grpcurl response contains a non-null `error` field despite a success status, add an override that nulls the `error` assertion: `"error": null`
5. Apply the same logic to any other suite where grpcurl or suite-runner output reveals status mismatches

Example override for a connector that returns `AUTHENTICATION_PENDING`:
```json
{
  "authorize": {
    "no3ds_auto_capture_credit_card": {
      "assert": {
        "status": {"one_of": ["CHARGED", "AUTHORIZED", "AUTHENTICATION_PENDING"]},
        "error": null
      }
    },
    "no3ds_manual_capture_credit_card": {
      "assert": {
        "status": {"one_of": ["AUTHORIZED", "AUTHENTICATION_PENDING"]},
        "error": null
      }
    }
  }
}
```

If `{CODEGEN_GRPCURL_OUTPUT}` is not available (standalone mode or `NOT_RUN`), skip override derivation. Existing overrides (if any) are preserved as-is.

## Phase 3: Edit the JSON Files

### 3a: `specs.json` format

Location: `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/specs.json`

```json
{
  "connector": "{connector}",
  "supported_suites": [
    "authorize",
    "capture",
    "void",
    "refund",
    "refund_sync",
    "get"
  ]
}
```

If the file does not exist, create it with the connector name and ALL suites that mapped from `{IMPLEMENTED_FLOWS}`.

If the file exists, add all new suite names to `supported_suites`. Never remove existing entries.

### 3b: `override.json` format

Location: `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/override.json`

Override only the fields that differ from the global baseline. Use JSON Merge Patch semantics:
- object keys merge recursively
- scalars and arrays replace existing values
- `null` removes a key

```json
{
  "<suite_name>": {
    "<scenario_name>": {
      "grpc_req": {
        "payment_method": {
          "card": {
            "card_number": { "value": "<connector-specific-test-card>" }
          }
        }
      },
      "assert": {
        "error.connector_details.message": { "contains": "<connector-specific-error-text>" }
      }
    }
  }
}
```

If the file does not exist and no overrides are needed, create it with an empty object: `{}`

If the file exists and currently contains `{}`, replace the entire content with the new override structure. Do not nest the override inside an empty object.

If the file exists and already has override entries, add the new suite/scenario override entries alongside the existing ones. Never remove existing entries unless fixing a known error.

### 3c: `scenario.json` format (only for Case B and Case C)

Location: `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/scenario.json`

Each scenario is a top-level key containing `grpc_req`, `assert`, and optionally `is_default`.

**WARNING**: Editing `scenario.json` in `global_suites/` is a global change that affects ALL connectors that support that suite. Use only generic, widely-valid test data. Put connector-specific values in `override.json` instead.

**CRITICAL**: Do NOT abbreviate or skip fields. You MUST read the first existing scenario in the same `scenario.json` file and replicate every field exactly, changing only `payment_method`, `description`, and `capture_method` as needed.

**CRITICAL**: When creating a Case B scenario for a new payment method, do NOT guess the `payment_method` JSON structure. Copy it from `{CODEGEN_GRPCURL_OUTPUT}` — the codegen agent already built a working grpcurl request with the correct `payment_method` block for the new payment method. Extract the `payment_method` object from the grpcurl `-d` request body and use it directly. Replace any real credentials or account numbers with generic test values or `"auto_generate"`.

Available assertion rules:
- `{ "must_exist": true }` — field must be present and non-null
- `{ "must_not_exist": true }` — field must be absent or null
- `{ "equals": <value> }` — field must equal the exact value
- `{ "one_of": [<value1>, <value2>] }` — field must match one of the values
- `{ "contains": "<substring>" }` — field string must contain the substring
- `{ "echo": "<label>" }` — prints the field value for debugging, does not assert

Optional field `is_default`:
- Set `"is_default": true` on exactly one scenario per suite to mark it as the default dependency target for downstream suites.
- Most suites already have a default scenario. Do NOT add `is_default` to a new scenario unless you are creating the first scenario in a new suite.

**For non-authorize suites** (capture, refund, void, etc.), the `grpc_req` shape is different. Always read the existing `scenario.json` in that suite first and replicate its exact field set.

### 3d: `suite_spec.json` format (only for Case C)

Location: `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/suite_spec.json`

**For independent suites** (no upstream dependencies):

```json
{
  "suite": "<flow_name>",
  "suite_type": "independent",
  "depends_on": [],
  "strict_dependencies": true
}
```

**For dependent suites** (need results from prior flows):

```json
{
  "suite": "<flow_name>",
  "suite_type": "dependent",
  "depends_on": [
    "create_access_token",
    "create_customer",
    {
      "suite": "authorize",
      "scenario": "no3ds_manual_capture_credit_card"
    }
  ],
  "dependency_scope": "scenario",
  "strict_dependencies": false
}
```

Fields:
- `suite` — suite name, must match the `<suite>` in the `<suite>_suite/` directory name
- `suite_type` — `"independent"` or `"dependent"`
- `depends_on` — list of upstream suites; each entry is either a plain suite name string or an object with `suite`, optional `scenario`, and optional `context_map`
- `strict_dependencies` — if `true`, a dependency failure stops this suite immediately
- `dependency_scope` — `"suite"` (default, dependencies run once) or `"scenario"` (dependencies run per-scenario). Most dependent suites on the repo use `"scenario"`. Omit this field for independent suites.

Use `capture_suite/suite_spec.json`, `refund_suite/suite_spec.json`, or `recurring_charge_suite/suite_spec.json` as reference for dependent suite patterns.

## Phase 4: Validate

You MUST run validation after editing any JSON files.

### 4a: Schema drift checks (always run these)

```bash
cargo test -p ucs-connector-tests all_supported_scenarios_match_proto_schema_for_all_connectors
```

```bash
cargo test -p ucs-connector-tests all_override_entries_match_existing_scenarios_and_proto_schema
```

These catch:
- scenario fields that do not exist in the proto request shape
- invalid enum values
- override entries pointing to missing suite/scenario keys
- typos in suite or scenario names

If either check fails, fix the JSON and rerun. You may fix JSON mistakes you introduced, then rerun. Do NOT change connector Rust code here.

### 4b: Run the suite for this connector (only when codegen grpc passed)

Only run the scenario harness when `{CODEGEN_GRPCURL_RESULT}` is `PASS`, because the harness spawns an in-process gRPC server and needs valid connector credentials.

**Set up the environment first:**

```bash
export CONNECTOR_AUTH_FILE_PATH="$PWD/creds.json"
```

If `creds.json` is not at the repo root, use the actual path to the credentials file.

**Then run all suites for this connector:**

```bash
cargo run -p ucs-connector-tests --bin suite_run_test -- --all --connector {connector}
```

Or run individual suites:

```bash
cargo run -p ucs-connector-tests --bin suite_run_test -- --suite <suite_name> --connector {connector}
```

If `{CODEGEN_GRPCURL_RESULT}` is `FAIL` or `NOT_RUN`, skip this step and rely on the schema drift checks alone for validation.

### 4c: Classify outcomes

- **PASS**: schema checks pass, and either the suite runner passes or the suite runner was skipped because grpc was not available
- **FAIL**: schema checks fail, or the suite runner fails with assertion errors or request errors that indicate a connector logic bug
- **NOT_SUPPORTED**: ALL implemented flows have no matching global suite (Case D for every flow), but `specs.json` / `override.json` files now exist

For non-`PASS` outcomes, produce a concrete repair hint for the next codegen attempt.

## Rules

1. **This phase is mandatory after every codegen attempt** — run it for both `SUCCESS` and `FAILED` codegen outputs.
2. **JSON only** — you edit `specs.json`, `override.json`, `scenario.json`, and `suite_spec.json`. You do NOT write Rust test code.
3. **Iterate over all flows** — do not stop at the first flow. Process every flow in `{IMPLEMENTED_FLOWS}` that has a matching global suite.
4. **Do not use real credentials** — use standard test values only. The `"auto_generate"` marker handles most fields.
5. **Do not change connector logic** — only test JSON files. If the scenario harness fails because connector logic is broken, return `FAIL` with a repair hint.
6. **Return actionable repair feedback** — every non-`PASS` result must include a useful `FAILURE_CATEGORY`, `FAILURE_FINGERPRINT`, and `REPAIR_HINT`.
7. **Do not commit** — the PR Agent handles all git operations later.
8. **Preserve existing entries** — never remove existing suites from `specs.json`, existing scenarios from `scenario.json`, or existing overrides from `override.json` unless fixing a known error you introduced.
9. **Schema checks are mandatory** — always run both `all_supported_scenarios_match_proto_schema_for_all_connectors` and `all_override_entries_match_existing_scenarios_and_proto_schema` after any JSON edit.
10. **Global scenario edits affect all connectors** — when adding a new scenario to `global_suites/`, use only generic test data. Put connector-specific values in `override.json`.
