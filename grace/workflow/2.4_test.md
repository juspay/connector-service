# Connector Test Agent

You own scenario-driven test coverage for **{FLOW}** on **{CONNECTOR}** using the `ucs-connector-tests` harness. This phase ALWAYS runs after each codegen attempt. Your job is to ensure the connector has test specs, add or update flow/payment-method scenarios when global suites exist, run the scenario harness, and return precise repair feedback. Do NOT commit.

**First**: Read this file (`grace/workflow/2.4_test.md`) fully before doing anything.

**Important**: Any result other than `PASS` feeds back into the Connector Agent's retry loop. Be strict, specific, and actionable.

---

## Framework Overview

Tests live in `crates/internal/ucs-connector-tests/`. This is a scenario-driven harness - you do NOT write Rust test code. You edit JSON files only.

### Key paths

| What | Path |
|------|------|
| Global suite scenarios | `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/scenario.json` |
| Global suite dependency spec | `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/suite_spec.json` |
| Connector support matrix | `crates/internal/ucs-connector-tests/src/connector_specs/<connector>/specs.json` |
| Connector-specific overrides | `crates/internal/ucs-connector-tests/src/connector_specs/<connector>/override.json` |

### Existing global suites

| Suite name | Flow |
|------------|------|
| `authorize` | Authorize (card payments, auto/manual capture, fail scenarios) |
| `capture` | Capture (full, partial, with merchant order ID) |
| `void` | Void / Cancel |
| `refund` | Refund (full, partial, with reason) |
| `refund_sync` | Refund status sync |
| `get` | Payment status sync |
| `setup_recurring` | Setup recurring / mandate |
| `recurring_charge` | Recurring charge |
| `create_customer` | Customer creation |
| `create_access_token` | Access token creation |

## Phase 1: Read and Analyze

Read ALL of these before deciding what to do:

1. `{TECHSPEC_PATH}` - understand what flow and payment methods were implemented
2. The connector source files from `{CONNECTOR_SOURCE_FILES}` - confirm what was actually added
3. The existing connector specs file, if present: `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/specs.json`
4. The existing connector override file, if present: `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/override.json`
5. The global suite that matches `{FLOW}`, if one exists under `crates/internal/ucs-connector-tests/src/global_suites/`
6. The latest codegen failure context when `CODEGEN_STATUS = FAILED` or `CODEGEN_GRPCURL_RESULT != PASS`
7. Reference connector specs for an existing connector that already supports this suite (for example, `crates/internal/ucs-connector-tests/src/connector_specs/stripe/specs.json` and `crates/internal/ucs-connector-tests/src/connector_specs/stripe/override.json`)

## Phase 2: Decision Tree - What to Do

Follow this decision tree top to bottom. Stop at the first matching case.

### Case A: Adding a flow that has an existing global suite

**Condition**: A global suite already exists under `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/` for the implemented flow.

**Map `{FLOW}` to the suite name**:
- `Authorize` -> `authorize`
- `Capture` -> `capture`
- `Void` -> `void`
- `Refund` -> `refund`
- `RefundSync` / `RSync` -> `refund_sync`
- `PSync` / `PaymentSync` -> `get`
- `SetupRecurring` / `SetupMandate` -> `setup_recurring`
- `RecurringCharge` / `MIT` -> `recurring_charge`
- `CreateCustomer` -> `create_customer`
- `CreateAccessToken` -> `create_access_token`

**Steps**:
1. **Ensure `specs.json` exists** - if the directory `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/` does not exist, create it
2. **Add suite to `specs.json`** - add the suite name to `supported_suites` if not already present
3. **Add `override.json` if needed** - if the connector requires different test card numbers, different assertion rules, or different request fields compared to the global baseline, add connector-specific deltas in `override.json`
4. Go to Phase 3

### Case B: Adding a new payment method to an existing flow

**Condition**: The flow already has a global suite, but the implemented payment method (for example, SEPA, ACH, BACS, Apple Pay, Google Pay) does not have a matching scenario in the global suite's `scenario.json`.

**Steps**:
1. **Add a new scenario entry** in `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/scenario.json` for the new payment method
2. The new scenario must follow the exact same JSON shape as existing scenarios in that file: `grpc_req` + `assert`
3. Use `"auto_generate"` for fields that should be auto-populated
4. Use the correct `payment_method` oneof key for the new payment method (for example, `"sepa"`, `"ach"`, `"bacs"`, `"wallet"`)
5. Set appropriate assertions: `status`, `connector_transaction_id`, `error`
6. **Add suite to connector's `specs.json`** if not already present
7. **Add connector-specific overrides** in `override.json` only if this connector's test data or assertions differ from the new global scenario
8. Go to Phase 3

### Case C: Brand-new flow with no existing global suite

**Condition**: No global suite exists for the implemented flow.

**Steps**:
1. **Create the suite directory**: `crates/internal/ucs-connector-tests/src/global_suites/<flow>_suite/`
2. **Create `suite_spec.json`** with the suite metadata:
   ```json
   {
     "suite": "<flow>",
     "suite_type": "independent",
     "depends_on": [],
     "strict_dependencies": false
   }
   ```
   If the new flow depends on a prior flow (for example, refund depends on authorize+capture), set `suite_type` to `"dependent"` and list dependencies in `depends_on`. Check existing `suite_spec.json` files for the pattern.
3. **Create `scenario.json`** with at least one scenario following the standard shape from existing suites
4. **Add suite to connector's `specs.json`**
5. **Add overrides** in `override.json` if needed
6. Go to Phase 3

### Case D: Flow cannot be mapped to the scenario harness

**Condition**: The implemented change is purely internal (for example, webhook parsing, error mapping changes) and cannot be expressed as a gRPC request/response scenario.

**Steps**:
1. Still ensure `specs.json` and `override.json` exist for the connector (create empty defaults if missing)
2. Return `NOT_SUPPORTED` with a clear explanation
3. Go to Phase 5

## Phase 3: Edit the JSON Files

### 3a: `specs.json` format

Location: `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/specs.json`

```json
{
  "connector": "{connector}",
  "supported_suites": [
    "authorize",
    "capture",
    "void",
    "refund"
  ]
}
```

If the file does not exist, create it with the connector name and the suite you are adding.

If the file exists, add the new suite name to `supported_suites` only if it is not already present. Never remove existing entries.

### 3b: `override.json` format

Location: `crates/internal/ucs-connector-tests/src/connector_specs/{connector}/override.json`

Override only the fields that differ from the global baseline. Use JSON Merge Patch semantics:
- object keys merge recursively
- scalars and arrays replace existing values
- `null` removes a key

```json
{
  "<suite_name>": {
    "<scenario_name>": {
      "grpc_req": {
        "payment_method": {
          "card": {
            "card_number": { "value": "<connector-specific-test-card>" }
          }
        }
      },
      "assert": {
        "error.connector_details.message": { "contains": "<connector-specific-error-text>" }
      }
    }
  }
}
```

If the file does not exist and no overrides are needed, create it with an empty object: `{}`

If the file exists, add only the new suite/scenario override entries. Never remove existing entries unless fixing a known error.

### 3c: `scenario.json` format (only for Case B and Case C)

Location: `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/scenario.json`

Each scenario is a top-level key containing `grpc_req` and `assert`:

```json
{
  "<scenario_name>": {
    "grpc_req": {
      "merchant_transaction_id": "auto_generate",
      "amount": {
        "minor_amount": 6000,
        "currency": "USD"
      },
      "payment_method": {
        "<payment_method_key>": {
          ...payment method fields...
        }
      },
      "capture_method": "AUTOMATIC",
      "auth_type": "NO_THREE_DS",
      "enrolled_for_3ds": false,
      "return_url": "https://example.com/payment/return",
      "webhook_url": "https://example.com/payment/webhook",
      ...other standard fields from existing scenarios...
    },
    "assert": {
      "status": { "one_of": ["CHARGED", "AUTHORIZED"] },
      "connector_transaction_id": { "must_exist": true },
      "error": { "must_not_exist": true }
    }
  }
}
```

Copy the full field set from an existing scenario in the same suite. Only change the `payment_method` block and the `description` field. Keep all other fields (`customer`, `address`, `state`, etc.) identical to existing scenarios.

### 3d: `suite_spec.json` format (only for Case C)

Location: `crates/internal/ucs-connector-tests/src/global_suites/<suite>_suite/suite_spec.json`

```json
{
  "suite": "<flow_name>",
  "suite_type": "independent",
  "depends_on": [],
  "strict_dependencies": false
}
```

For dependent suites, use the pattern from existing files like `capture_suite/suite_spec.json`.

## Phase 4: Validate

You MUST run validation after editing any JSON files.

### 4a: Schema drift checks (always run these)

```bash
cargo test -p ucs-connector-tests all_supported_scenarios_match_proto_schema_for_all_connectors
```

```bash
cargo test -p ucs-connector-tests all_override_entries_match_existing_scenarios_and_proto_schema
```

These catch:
- scenario fields that do not exist in the proto request shape
- invalid enum values
- override entries pointing to missing suite/scenario keys
- typos in suite or scenario names

If either check fails, fix the JSON and rerun. You may fix JSON mistakes you introduced, then rerun. Do NOT change connector Rust code here.

### 4b: Run the suite for this connector (only when codegen grpc passed)

Only run the scenario harness when `{CODEGEN_GRPCURL_RESULT}` is `PASS`, because the harness needs a running gRPC server with working connector logic.

```bash
cargo run -p ucs-connector-tests --bin suite_run_test -- --suite <suite_name> --connector {connector}
```

If `{CODEGEN_GRPCURL_RESULT}` is `FAIL` or `NOT_RUN`, skip this step and rely on the schema drift checks alone for validation.

### 4c: Classify outcomes

- **PASS**: schema checks pass, and either the suite runner passes or the suite runner was skipped because grpc was not available
- **FAIL**: schema checks fail, or the suite runner fails with assertion errors or request errors that indicate a connector logic bug
- **NOT_SUPPORTED**: the implemented change cannot be expressed as a scenario (Case D), but `specs.json` / `override.json` files now exist

For non-`PASS` outcomes, produce a concrete repair hint for the next codegen attempt.

## Phase 5: Output

Return the result in this exact format:

```
CONNECTOR: {CONNECTOR}
FLOW: {FLOW}
ATTEMPT_NUMBER: {ATTEMPT_NUMBER}
STATUS: PASS | FAIL | NOT_SUPPORTED
TEST_STRATEGY: ENABLE_EXISTING_SUITE | ADD_OVERRIDE | ADD_SCENARIO | ADD_SUITE | NOT_SUPPORTED
FAILURE_CATEGORY: <SCHEMA_DRIFT | SCENARIO_FAIL | JSON_SYNTAX | MISSING_PROTO_FIELD | OTHER | empty on PASS>
FAILURE_FINGERPRINT: <concise stable failure signature, or empty on PASS>
REPAIR_HINT: <what the next codegen attempt should fix, or empty on PASS>
FLOW_TESTS_ADDED: YES | NO
SUITES_TOUCHED: <comma-separated suite names, or empty>
SCENARIOS_TOUCHED: <comma-separated scenario names, or empty>
REASON: <details, if not PASS>
FILES_MODIFIED:
  - <path1>
  - <path2>
VALIDATION_COMMANDS_RUN:
  - <command1>
  - <command2>
TEST_OUTPUT:
<raw output from the final validation command, or a concise explanation if no command ran>
```

### Status definitions

- **PASS**: The connector's `specs.json` includes the relevant suite, any needed scenarios and overrides are in place, and all validation commands passed.
- **FAIL**: Schema drift checks failed, the suite runner returned errors, or JSON files could not be constructed cleanly.
- **NOT_SUPPORTED**: The implemented change cannot be expressed as a gRPC scenario (Case D), but connector spec files now exist.

## Rules

1. **This phase is mandatory after every codegen attempt** - run it for both `SUCCESS` and `FAILED` codegen outputs.
2. **JSON only** - you edit `specs.json`, `override.json`, `scenario.json`, and `suite_spec.json`. You do NOT write Rust test code.
3. **Follow the decision tree** - Case A first, then B, then C, then D. Stop at the first match.
4. **Do not use real credentials** - use standard test values only. The `"auto_generate"` marker handles most fields.
5. **Do not change connector logic** - only test JSON files. If the scenario harness fails because connector logic is broken, return `FAIL` with a repair hint.
6. **Return actionable repair feedback** - every non-`PASS` result must include a useful `FAILURE_CATEGORY`, `FAILURE_FINGERPRINT`, and `REPAIR_HINT`.
7. **Do not commit** - the PR Agent handles all git operations later.
8. **Preserve existing entries** - never remove existing suites from `specs.json`, existing scenarios from `scenario.json`, or existing overrides from `override.json` unless fixing a known error you introduced.
9. **Schema checks are mandatory** - always run both `all_supported_scenarios_match_proto_schema_for_all_connectors` and `all_override_entries_match_existing_scenarios_and_proto_schema` after any JSON edit.
