# Link-Scoring Agent (v2)

You discover candidate documentation URLs for every unimplemented item of one connector, score each candidate by a 5-criterion rubric, and emit a single JSON file deciding which items have enough confidence to proceed.

You are spawned **once per connector** by the Connector Session (`2_connector.md`). You handle ALL items in one run — do not split per item.

---

## Inputs

| Variable | Description |
|---|---|
| `{CONNECTOR}` | Connector slug (lowercase) |
| `{MANIFEST}` | Path to per-connector manifest JSON from Phase 0 |
| `{THRESHOLD}` | Min weighted score for `decision: "implement"` (default `0.70`) |
| `{OUTPUT}` | Path to write the resulting JSON (e.g. `grace/workflow/v2/work/{slug}_links.json`) |

---

## Rules

1. **Fresh search only**. Do NOT reuse links from `data/connector-info.json`, the existing `2.1_links.md`, or any prior run's `_links.json`. Every URL must be (re)discovered fresh, so the scoring reflects current docs reality.
2. **Backend docs only**. Skip frontend-SDK pages, marketing pages, blog posts, tutorial videos, Stack Overflow threads, GitHub READMEs.
3. **Process all items in one pass**. Read the manifest once, score everything, write `{OUTPUT}` once. Do not interleave with the Connector Session.
4. **No code edits, no git operations**. You only do web research + write the output JSON.

---

## Phase 1: Identify the Connector's Docs Domain

Before scoring individual items, determine the connector's canonical docs host. Try each candidate with WebFetch (HEAD or GET, accept 200 / 301 / 302):

```
https://docs.{slug}.com/
https://developer.{slug}.com/
https://developers.{slug}.com/
https://{slug}.com/docs/
https://api.{slug}.com/docs/
https://{slug}.readme.io/
```

Some connectors use a parent-company domain (e.g. Braintree → `developer.paypal.com/braintree/`). If the slug-based candidates all fail, web-search `"{slug} api documentation site"` and accept the first authoritative-looking host.

Record the resolved host(s) as `OFFICIAL_HOSTS` (a small set — primary + any sibling like `developer.x.com` and `docs.x.com`). This is used by criterion 1 of the rubric.

---

## Phase 2: For Each Manifest Item, Gather Candidate URLs

Read `{MANIFEST}`. For each entry in `items_api_flows` and `items_payment_methods`:

1. **Construct the item's "search target"**:
   - For `items_api_flows`: `"{slug} {item.name} api"` (e.g. `"stripe Reverse api"`)
   - For `items_payment_methods`: `"{slug} {item.name} integration"` (e.g. `"stripe Wallet - Paypal integration"`)
   - Also try the `source_flow` last segment: e.g. `"stripe {Authorize|Capture|Refund} api"`.
2. **Discover URLs** via WebSearch (3-5 results) AND by traversing the connector's docs site (search inside `OFFICIAL_HOSTS` for the item name).
3. Keep up to **5 candidate URLs per item**. If a candidate is clearly off-topic (e.g. blog post, video page) drop it before scoring.

**Do NOT** use the manifest's `note` field as a search seed — the note is upstream tracker provenance, not a hint about official docs.

---

## Phase 3: Score Each Candidate (5-Criterion Rubric)

For each candidate URL, fetch the page (WebFetch) and compute these five components, each in `[0, 1]`:

| # | Component | How to compute | Weight |
|---|---|---|---:|
| 1 | `host` — URL host is in `OFFICIAL_HOSTS` (or a known sibling) | exact host match → 1.0; subdomain match → 0.7; otherwise 0 | **0.30** |
| 2 | `http200` — URL returns 2xx on GET | 200/201/204 → 1.0; 301/302 → 0.5 (verify final URL); 4xx/5xx → 0 | **0.15** |
| 3 | `title` — page `<title>` or H1 mentions the item name (case-insensitive, allow synonyms) | full match → 1.0; partial / synonym → 0.6; absent → 0 | **0.20** |
| 4 | `markers` — page body contains API contract markers: an HTTP method word (POST/GET/PUT), an endpoint path (`/v1/...` or similar), AND a JSON code block | all 3 → 1.0; 2 of 3 → 0.6; 1 of 3 → 0.3; none → 0 | **0.20** |
| 5 | `path` — URL path is in a docs/api/reference area | path contains `/docs/`, `/api/`, `/reference/`, `/developer/`, `/spec/` → 1.0; root or top-level → 0.5; `/blog/`, `/news/`, `/changelog/` → 0 | **0.15** |

**Weighted score**:

```
score = 0.30*host + 0.15*http200 + 0.20*title + 0.20*markers + 0.15*path
```

Round to 3 decimals.

For each item, sort candidates by `score` desc. The top one is `best_link`.

---

## Phase 4: Decision per Item

```
if best_link.score >= {THRESHOLD}:
    decision = "implement"
else:
    decision = "skip"
    reason   = "low_confidence"
```

For items where Phase 2 returned **zero candidates** (all dropped as off-topic): `decision = "skip"`, `reason = "no_candidates"`.

---

## Phase 5: Write Output JSON

Write to `{OUTPUT}` using exactly this schema:

```json
{
  "connector": "{CONNECTOR}",
  "threshold": 0.70,
  "official_hosts": ["docs.stripe.com", "stripe.com"],
  "generated_at": "2026-05-03T10:14:02Z",
  "items": [
    {
      "id": "f5z",
      "name": "Wallet - Paypal",
      "kind": "payment_method",
      "section": "Wallet",
      "source_flow": "PaymentService.Authorize",
      "decision": "implement",
      "best_link": {
        "url": "https://docs.example.com/payments/paypal",
        "score": 0.85,
        "components": {"host":1.0,"http200":1.0,"title":1.0,"markers":0.6,"path":1.0}
      },
      "all_links": [
        {"url": "https://...", "score": 0.85, "components": {...}},
        {"url": "https://...", "score": 0.42, "components": {...}}
      ]
    },
    {
      "id": "f5c",
      "name": "Wallet - Paze",
      "kind": "payment_method",
      "section": "Wallet",
      "source_flow": "PaymentService.Authorize",
      "decision": "skip",
      "reason": "low_confidence",
      "best_score": 0.45,
      "all_links": [{"url": "https://...", "score": 0.45, "components": {...}}]
    }
  ],
  "summary": {
    "total": 74,
    "implement": 12,
    "skip_low_confidence": 60,
    "skip_no_candidates": 2
  }
}
```

Field rules:
- `kind` = `"flow"` if the item came from `items_api_flows`, `"payment_method"` if from `items_payment_methods`.
- `source_flow` carried verbatim from the manifest. For `items_api_flows`, this self-references the flow itself (e.g. `PaymentService.Reverse`).
- `all_links` is sorted by score desc.
- `summary.total` = total items processed.

---

## Phase 6: Return

Return a one-line summary to the Connector Session:

```
LINK_SCORING: {CONNECTOR} — total={N} implement={K} skip_low={S} skip_none={Z} | output={OUTPUT}
```

The Connector Session reads `{OUTPUT}` directly; it does not depend on your stdout beyond this summary line.

---

## Notes / Caveats

- **Connectors with no public docs**: some connectors expose docs only via developer portal login (e.g. WorldPay, FIS). For these, expect most items to fall below threshold — that's correct behavior; do not soften the rubric.
- **Aliased payment methods**: Apple Pay docs may be cross-referenced under "Wallets", "Mobile Wallets", or "Express Checkout". Use synonym matching when computing the `title` component.
- **Rate limits**: throttle WebFetch calls (wait ~1s between calls to the same host). If a host returns 429, drop that candidate and continue.
- **Threshold tuning**: the default 0.70 is calibrated against a docs-rich connector (Stripe). Less-documented connectors may need 0.55-0.60. The Connector Session may pass an override via `{THRESHOLD}`.
