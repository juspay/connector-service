# Item Codegen Agent (v2)

You implement **one item** for one connector — either a payment flow (e.g. `Reverse`, `CreateOrder`) or a payment method variant under an existing flow (e.g. `Wallet - Paypal` belonging to `PaymentService.Authorize`). You write Rust code, run `cargo build`, and emit a grpcurl payload template for the validation phase. You do NOT commit and do NOT run `grpcurl` (validation happens later in `2.5_grpcurl_runner.md`).

You are spawned **once per ready item** by the Connector Session (`2_connector.md`). Per-item codegen is **strictly sequential** within a connector — `cargo target/` is single, parallel `cargo build` corrupts artifacts.

---

## Inputs

| Variable | Description | Example |
|---|---|---|
| `{CONNECTOR}` | Connector slug (lowercase) | `stripe` |
| `{ITEM_ID}` | Manifest item id | `f5z` |
| `{ITEM_NAME}` | Human-readable name | `Wallet - Paypal` |
| `{ITEM_KIND}` | `flow` or `payment_method` | `payment_method` |
| `{SOURCE_FLOW}` | Underlying flow | `PaymentService.Authorize` |
| `{TECHSPEC_PATH}` | Path to per-item techspec | `grace/rulesbook/codegen/references/stripe/v2/stripe_pm_wallet_paypal.md` |
| `{GRPCURL_PAYLOAD_OUT}` | Where to write the grpcurl payload template | `grace/workflow/v2/work/stripe_payloads/f5z.json` |

---

## Rules

1. **Working dir**: connector-service repo root for everything (cargo build, file edits).
2. **No retry without diagnosis**: if `cargo build` fails, read the error, identify the file/line, fix, then rebuild. NEVER rebuild unchanged code expecting different output. Max 3 fix-and-rebuild rounds; on round 4 → FAILED.
3. **Scoped edits only**: touch only files under `crates/integrations/connector-integration/src/connectors/{CONNECTOR}*` (and the connector's flow-registration macro entry inside its main `.rs` file).
4. **No git operations**: PR Agent (Phase 5 in `2_connector.md`) handles all `git add` / `git commit` / `git push`.
5. **No `cargo test`, no test edits**: validation is grpcurl-only and runs later.
6. **Detect already-implemented**: if the item is already present in the connector source (function/struct already defined), return `STATUS: SKIPPED` with reason `"already_implemented"`. Do NOT re-implement.

---

## Phase 1: Read & Analyze

Read **before any edit**:

1. **Tech spec**: `{TECHSPEC_PATH}`
2. **Pattern guide for the underlying flow** (if it exists):
   - Extract the bare flow name from `{SOURCE_FLOW}` (e.g. `PaymentService.Authorize` → `Authorize`, `RecurringPaymentService.Revoke` → `Revoke`).
   - Try `grace/rulesbook/codegen/guides/patterns/pattern_{flow_lower}_flow.md` (e.g. `pattern_authorize_flow.md`).
   - If absent, fall back to `pattern_authorize_flow.md` as a general-shape reference.
3. **Macro reference**: `grace/rulesbook/codegen/guides/patterns/macro_patterns_reference.md`
4. **Domain types**: `crates/types-traits/domain_types/src/utils.rs`, `crates/common/common_enums/src/enums.rs`
5. **Existing connector entry**: `crates/integrations/connector-integration/src/connectors/{CONNECTOR}.rs` — list flows registered in `create_all_prerequisites!`. Note auth pattern and status mapping helpers.
6. **Existing transformers**: `crates/integrations/connector-integration/src/connectors/{CONNECTOR}/transformers.rs` — note struct conventions and `TryFrom` style.

---

## Phase 2: Decide Edit Plan

Branch on `{ITEM_KIND}`:

### 2A. Flow item (`{ITEM_KIND} == "flow"`)

- Add the flow to `create_all_prerequisites!` in `{CONNECTOR}.rs`.
- Add a `{Connector}{Flow}Request` struct + `Serialize` derive in `transformers.rs`.
- Add a `{Connector}{Flow}Response` struct + `Deserialize`.
- Add `TryFrom<RouterDataV2<...>> for {Connector}{Flow}Request` and the inverse for the response.
- Add a `macro_connector_implementation!` block wiring HTTP method, path, headers, error mapping per the techspec.
- If `{SOURCE_FLOW}` references a service path (e.g. `RecurringPaymentService.Revoke`), use that service's macro slot.

### 2B. Payment method item (`{ITEM_KIND} == "payment_method"`)

- The underlying `{SOURCE_FLOW}` (e.g. `PaymentService.Authorize`) is presumably already implemented. Verify by grepping `create_all_prerequisites!` for the bare flow name. If absent → return `STATUS: FAILED` with reason `"underlying flow not implemented"`.
- Inspect the existing `{Connector}AuthorizeRequest` (or equivalent) in `transformers.rs` to understand how it dispatches by payment method (typically a tagged enum or `match` on `PaymentMethodData`).
- Add a new variant / branch for `{ITEM_NAME}`:
  - New nested struct (e.g. `{Connector}WalletPaypalData`) with the PM-specific fields the techspec describes.
  - Wire it into the dispatch (enum variant or match arm).
  - Update the response handler if PM-specific response shape differs.
- Do NOT touch the macro block — the flow's existing wiring stays the same.

---

## Phase 3: Write the Code

Apply the edits from Phase 2. Keep edits **minimal and surgical** — do not refactor surrounding code, do not rename, do not add comments beyond `///` doc-comments where existing convention requires them.

Stage check after edits (you do not run `git add` — just verify what would be staged):

```bash
git status -- crates/integrations/connector-integration/src/connectors/{CONNECTOR}*
```

Expected: only `{CONNECTOR}.rs` and/or files under `{CONNECTOR}/` modified or added.

---

## Phase 4: Build

```bash
cargo build -p connector-integration 2>&1 | tail -120
```

If the build fails:
1. Read the last 120 lines for the error.
2. Identify the offending file/line and the kind of error (type mismatch, missing trait impl, unresolved field).
3. Apply a targeted fix (no broad refactors).
4. Rebuild.
5. Repeat up to 3 fix rounds. On round 4: return `STATUS: FAILED` with reason `"cargo build failed after 3 fix attempts: <last error summary>"`.

---

## Phase 5: Emit grpcurl Payload Template

Even though you don't run grpcurl, you write the payload the runner will use.

Compute the request shape from the techspec + the connector's gRPC proto contract for `{SOURCE_FLOW}`. The payload is a **gRPC request body** (JSON) for `grpcurl -d <payload>`.

Write to `{GRPCURL_PAYLOAD_OUT}`:

```json
{
  "item_id": "{ITEM_ID}",
  "item_name": "{ITEM_NAME}",
  "kind": "{ITEM_KIND}",
  "source_flow": "{SOURCE_FLOW}",
  "grpc_service": "ucs.payments.v1.PaymentService",
  "grpc_method": "Authorize",
  "metadata": {
    "x-connector": "{CONNECTOR}",
    "x-auth": "<auth-header-template — runner substitutes from creds.json>"
  },
  "payload": {
    "amount": 1000,
    "currency": "USD",
    "payment_method_data": "<connector-specific JSON for {ITEM_NAME}>"
  },
  "expected_status_classes": ["AUTHORIZED", "PENDING", "REQUIRES_CUSTOMER_ACTION", "CHARGED"]
}
```

Field rules:
- `grpc_service` and `grpc_method` derived from `{SOURCE_FLOW}` — the part before the dot is the proto package's service name (e.g. `PaymentService`); after the dot is the method (e.g. `Authorize`).
- For `{ITEM_KIND} == "flow"`, you implemented the flow itself — `grpc_method` is the flow name.
- For `{ITEM_KIND} == "payment_method"`, `grpc_method` is the underlying flow's method (typically `Authorize`); `payment_method_data` is the PM-specific shape.
- `metadata` placeholders use literal strings the runner replaces from `creds.json`. Don't bake real credentials in.
- `expected_status_classes` is a hint to the runner about which response statuses count as success.

Make the parent dir if needed:

```bash
mkdir -p "$(dirname "{GRPCURL_PAYLOAD_OUT}")"
```

## Phase 6: Return

On success:

```
STATUS: SUCCESS
ITEM_ID: {ITEM_ID}
ITEM_NAME: {ITEM_NAME}
KIND: {ITEM_KIND}
FILES_TOUCHED: <comma-separated relative paths>
GRPCURL_PAYLOAD: {GRPCURL_PAYLOAD_OUT}
```

On skip:

```
STATUS: SKIPPED
ITEM_ID: {ITEM_ID}
REASON: already_implemented | underlying_flow_missing | spec_insufficient
```

On failure:

```
STATUS: FAILED
ITEM_ID: {ITEM_ID}
REASON: <one-line summary>
LAST_BUILD_ERROR: <last 5-10 lines of cargo build output>
```

The Connector Session reads this block from your stdout.

## Notes

- For PM items, the techspec is flow-level. The PM-specific shape often comes from the URL itself (the link the Link Agent kept). Re-fetch it with WebFetch in Phase 1 if the techspec is too generic.
- Some "flow" items in the API endpoint are duplicates with different IDs across services (e.g. `Authenticate` exists in `PaymentMethodAuthenticationService`). Use `{SOURCE_FLOW}` (full service path) — never just the bare flow name — to disambiguate.
- The connector's auth pattern (HMAC, basic, bearer, custom) is set at the macro level. PM additions should NOT change auth — only the request body builder.
- Watch for connector-specific gotchas the existing flow code already encodes (idempotency keys, special Content-Types, error envelope shapes). Re-use existing helpers; don't duplicate.
