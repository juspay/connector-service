# PR Agent (v2)

You handle **all git operations** for one connector's v2 batch — committing per-item changes on the dev branch, cherry-picking onto a clean per-connector PR branch, scrubbing credentials, pushing, and opening a PR on `juspay/connector-service`.

This is a thin wrapper over the existing v1 PR Agent (`grace/workflow/2.4_pr.md`). The differences are:

1. **Multiple items per connector**, not a single flow. The dev-branch commit list contains one commit per ITEM (not per connector).
2. **Inputs come from a results JSON**, not from per-item args. The agent reads `{RESULTS}` and groups by item.
3. **PR description aggregates per-item grpcurl output**, not a single flow's output.

You are spawned **once per connector** by the Connector Session (`2_connector.md`) at the end of Phase 5.

---

## Inputs

| Variable | Description |
|---|---|
| `{CONNECTOR}` | Connector slug (lowercase) |
| `{DEV_BRANCH}` | Shared dev branch where per-item commits land first |
| `{RESULTS}` | Path to the partially-populated `{CONNECTOR}_results.json` |
| `{ITEMS_FOR_PR}` | JSON array of items eligible for PR inclusion (`grpcurl_status` is `SUCCESS` or `FAILED`; `SKIPPED` items are excluded) |

---

## Rules

1. **HARD GUARDRAIL — TARGET REPO**: ALL `gh` commands MUST use `--repo juspay/connector-service`. No other repo.
2. **HARD GUARDRAIL — NO retry of identical commands**: if `gh pr create` or `git push` fails, diagnose and fix; never retry the same command.
3. **No `git add -A`**: stage only the connector's source files per item.
4. **Credential scrub**: before pushing, run the scrubber (Phase 4 below). Any commit containing a value present in `creds.json` is rejected.
5. **Same-repo PR**: `origin` is `juspay/connector-service`. Do NOT use cross-fork `head:` syntax.
6. **One PR per connector** for the whole v2 batch — not one PR per item. The PR title summarizes the batch; the body lists per-item statuses.

---

## Phase 1: Per-item Commits on `{DEV_BRANCH}`

Verify branch:

```bash
git branch --show-current   # must be {DEV_BRANCH}
```

For each item in `{ITEMS_FOR_PR}` (in order), stage the item's touched files and commit:

```bash
# Files come from the results JSON entry's `files_touched`
git add <files for this item>
git commit -m "feat({CONNECTOR}): implement {item.name} ({item.id})

Source flow: {item.source_flow}
Kind: {item.kind}
grpcurl: {item.grpcurl_status}
"
```

If `git add` shows zero files staged for an item (codegen wrote nothing), skip its commit and continue.

After processing all items, capture the per-item commit SHAs in order:

```bash
git log --format='%H %s' "{DEV_BRANCH}~$(jq '. | length' <<<'{ITEMS_FOR_PR}')..{DEV_BRANCH}"
```

---

## Phase 2: Cherry-Pick onto Clean PR Branch

PR-branch convention: `feat/grace-v2-{CONNECTOR}-$(date -u +%Y%m%d)`.

```bash
PR_BRANCH="feat/grace-v2-{CONNECTOR}-$(date -u +%Y%m%d)"
git fetch origin main
git checkout -B "$PR_BRANCH" origin/main
# cherry-pick each captured SHA in order
for sha in <commit SHAs>; do
  git cherry-pick "$sha" || { git cherry-pick --abort; exit 1; }
done
```

If any cherry-pick conflicts: report `STATUS: FAILED`, reason `"cherry-pick conflict on <sha>"`. Don't try to auto-resolve.

---

## Phase 3: Credential Scrub

Run a content scan on the diff against `creds.json` values:

```bash
git diff origin/main..HEAD > /tmp/v2_pr_diff.patch
jq -r 'paths(scalars) as $p | getpath($p) | select(type=="string" and length > 8)' creds.json \
    > /tmp/v2_creds_values.txt
matches=$(grep -F -f /tmp/v2_creds_values.txt /tmp/v2_pr_diff.patch || true)
[[ -n "$matches" ]] && { echo "credentials in diff:"; echo "$matches"; exit 1; }
```

If any matches: report `STATUS: FAILED`, reason `"credential leak detected"`. Do NOT push.

---

## Phase 4: Push & Open PR

```bash
git push -u origin "$PR_BRANCH"
```

Build PR title:

```
feat({CONNECTOR}): grace v2 batch — {N_success} success, {N_failed} failed
```

Build PR body using the template below. `do-not-merge` label applies if any item is FAILED.

````markdown
## Summary

Generated by the **Grace v2 workflow** ([orchestrator](grace/workflow/v2/1_orchestrator.md) → [connector session](grace/workflow/v2/2_connector.md)).

- Connector: **{CONNECTOR}**
- Items implemented: {N_success}
- Items failed: {N_failed}
- Items skipped (low confidence / techspec missing / already implemented): {N_skipped}

## Per-item Status

| ID | Name | Kind | Source Flow | codegen | grpcurl | Status |
|---|---|---|---|---|---|---|
| {item.id} | {item.name} | {item.kind} | {item.source_flow} | {item.codegen_status} | {item.grpcurl_status} | {item.status} |

## grpcurl Output (per item)

<details>
<summary>{item.id} — {item.name}</summary>

```
{item.grpcurl_command_and_response}
```

</details>

## Notes

- Confidence threshold used: {threshold}
- Items below threshold are NOT included in this PR; see `grace/workflow/v2/work/{CONNECTOR}_links.json` for the skip list.
- This PR is on the same dev branch family as v1 PRs but uses the `feat/grace-v2-*` prefix so they do not collide.
````

```bash
LABEL_FLAGS=""
[[ "$N_FAILED" -gt 0 ]] && LABEL_FLAGS="--label do-not-merge"

gh pr create \
    --repo juspay/connector-service \
    --base main \
    --head "$PR_BRANCH" \
    --title "<title>" \
    --body-file /tmp/v2_pr_body.md \
    $LABEL_FLAGS
```

Capture `PR_URL` from the gh output.

---

## Phase 5: Return to Dev Branch

```bash
git checkout {DEV_BRANCH}
```

---

## Phase 6: Return

On success:

```
STATUS: SUCCESS
PR_URL: <url>
PR_BRANCH: <branch>
COMMITS: <comma-separated SHAs>
```

On failure:

```
STATUS: FAILED
PR_URL:
REASON: <one-line>
```

The Connector Session writes the PR_URL into `{CONNECTOR}_results.json` and reports back to the Orchestrator.
