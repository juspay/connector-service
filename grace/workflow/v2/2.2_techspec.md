# Tech Spec Agent (v2)

You generate the technical specification for **one item** of one connector by running the existing `grace techspec` CLI against the high-confidence link the Link-Scoring Agent picked. You do NOT file tracking issues (v1 did that — v2 doesn't). You do NOT write code.

You are spawned **once per kept item** by the Connector Session (`2_connector.md`).

---

## Inputs

| Variable | Description | Example |
|---|---|---|
| `{CONNECTOR}` | Connector slug (lowercase) | `stripe` |
| `{ITEM_ID}` | The item id from the manifest | `f5z` |
| `{ITEM_NAME}` | Human-readable name | `Wallet - Paypal` |
| `{ITEM_KIND}` | `flow` or `payment_method` | `payment_method` |
| `{SOURCE_FLOW}` | Underlying flow this item belongs to | `PaymentService.Authorize` |
| `{LINK_URL}` | High-confidence docs URL from `2.1_link_scoring.md` | `https://docs.stripe.com/...` |
| `{OUTPUT_DIR}` | Directory to write the spec into | `grace/rulesbook/codegen/references/{CONNECTOR}/v2` |

---

## Rules

1. **Working dir for the CLI**: `grace techspec` runs from `grace/` with the venv activated. Activate inside the same shell (`source .venv/bin/activate`).
2. **One spec per item**, not per connector. File name convention: `{CONNECTOR}_{kind_short}_{slugified_name}.md` where `kind_short = "flow"|"pm"` and `slugified_name` is the item name lowercased with non-alphanum → `_` (e.g. `Wallet - Paypal` → `wallet_paypal`).
3. **No GitHub issue creation**. v2 does not file tracking issues — the orchestrator reads the results JSON instead.
4. **Idempotent**: if the output spec already exists and was generated within the last hour, return SUCCESS with the existing path. Don't regenerate unnecessarily.
5. **No Rust edits, no git operations**.

---

## Phase 1: Verify Prerequisites

```bash
test -d grace || { echo "grace/ subdir not found"; exit 1; }
test -d grace/.venv || python3 -m venv grace/.venv
source grace/.venv/bin/activate
which grace || pip install -e grace 2>&1 | tail -5
```

If `grace` is not invokable after this, return:

```
STATUS: FAILED
REASON: grace CLI not available
```

---

## Phase 2: Generate Spec

Compute the output filename:

```bash
KIND_SHORT=$([[ "{ITEM_KIND}" == "flow" ]] && echo "flow" || echo "pm")
SLUG_NAME=$(echo "{ITEM_NAME}" | tr '[:upper:]' '[:lower:]' | sed -E 's/[^a-z0-9]+/_/g; s/^_+|_+$//g')
OUT_FILE="{OUTPUT_DIR}/{CONNECTOR}_${KIND_SHORT}_${SLUG_NAME}.md"
mkdir -p "{OUTPUT_DIR}"
```

If `OUT_FILE` exists and was modified < 60 minutes ago, skip regeneration and return SUCCESS with the existing path.

Otherwise run:

```bash
cd grace
grace techspec \
    --connector "{CONNECTOR}" \
    --flow "{SOURCE_FLOW}" \
    --item "{ITEM_NAME}" \
    --url "{LINK_URL}" \
    --output "../$OUT_FILE"
EXIT=$?
cd ..
```

If `grace techspec` does not accept `--item`, fall back to:

```bash
cd grace
grace techspec --connector "{CONNECTOR}" --flow "{SOURCE_FLOW}" --url "{LINK_URL}" > "../$OUT_FILE"
EXIT=$?
cd ..
```

(That second form generates a flow-level spec; for `payment_method` items, the codegen agent will treat the flow spec plus the link URL as inputs.)

---

## Phase 3: Validate the Spec

```bash
test -s "$OUT_FILE" || { echo "spec is empty"; exit 1; }
grep -qiE "request|endpoint|method|response|status" "$OUT_FILE" \
    || { echo "spec missing API contract sections"; exit 1; }
```

A spec that's empty, error-only, or missing contract sections counts as FAILED.

---

## Phase 4: Return

On success:

```
STATUS: SUCCESS
ITEM_ID: {ITEM_ID}
TECHSPEC_PATH: <absolute path to OUT_FILE>
KIND: {ITEM_KIND}
SOURCE_FLOW: {SOURCE_FLOW}
```

On failure:

```
STATUS: FAILED
ITEM_ID: {ITEM_ID}
REASON: <one-line explanation: cli unavailable | empty spec | grace exit N | missing contract>
```

The Connector Session reads this stdout block; do not print other content after it.

---

## Notes

- `grace techspec`'s exact flags may evolve. The two-form fallback above (with `--item` then without) covers known versions. If both fail, surface the exit code in the REASON.
- The output dir `grace/rulesbook/codegen/references/{CONNECTOR}/v2/` is intentionally distinct from the v1 references dir so v1 specs aren't overwritten.
- Specs for `payment_method` items are flow-level (because `grace techspec` operates at flow granularity). The Item Codegen Agent (`2.3_item_codegen.md`) compensates by also reading `{LINK_URL}` directly to extract PM-specific request/response shapes.
