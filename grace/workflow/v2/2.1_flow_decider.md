### How to Detect "Actually Implemented" vs "Stub/Default"

A connector MUST implement all flow traits (Rust's type system requires it), but most are **stubs**. A flow is **actually implemented** if and only if:

1. The connector's `impl ConnectorIntegrationV2<FlowMarker, ...>` block **overrides `get_url()`** to return a non-empty URL string.
2. Alternatively, the flow appears in the connector's `create_all_prerequisites!` macro invocation with a `request_body` and/or `response_body` — this means the connector has real request/response transformers for that flow.

A flow is a **stub** if:
- The `impl ConnectorIntegrationV2<FlowMarker, ...> for ConnectorName<T> {}` block is **empty** (no method overrides).
- Or the `get_url()` returns `Ok(String::new())` (empty string).

**Quick heuristic:** Look at the `create_all_prerequisites!` macro call in the connector's `.rs` file. The flows listed in the `api: [...]` array are the actually implemented flows. Everything else is a stub.

### Prerequisite Flow Gating (ValidationTrait)

Some flows are prerequisite steps that only execute if the connector opts in:

| Method | Default | Meaning |
|--------|---------|---------|
| `should_do_order_create()` | `false` | Create order before authorize |
| `should_do_session_token()` | `false` | Create session token before authorize |
| `should_do_access_token()` | `false` | Get OAuth access token before authorize |
| `should_create_connector_customer()` | `false` | Create customer before authorize |
| `should_do_payment_method_token()` | `false` | Tokenize payment method before authorize |

Check the connector's `impl ValidationTrait for ConnectorName<T>` block. If a method is overridden to return `true`, that prerequisite flow is active.

### What It Does

This is the **Hyperswitch Connector Generator v2.0** — a Bash script that scaffolds a new connector's foundation in connector-service. It:

1. **Creates 2 new files:**
   - `crates/integrations/connector-integration/src/connectors/{name}.rs` — Main connector file with struct, `ConnectorCommon`, `BodyDecoding`, and **empty stub implementations for ALL flow traits**
   - `crates/integrations/connector-integration/src/connectors/{name}/transformers.rs` — Auth type, error response struct, basic payment request/response types with `TryFrom` transformers

2. **Modifies 10+ existing files:**
   - Protobuf: Adds connector enum variant, config message, and oneof entry to `payment.proto`
   - Domain types: Adds to `ConnectorEnum`, gRPC-to-enum mapping, and `ConnectorParams`
   - Router data: Adds `ConnectorSpecificConfig` variant, auth conversion, macro entries
   - Connector module: Adds `pub mod {name}` and re-export
   - Integration types: Adds `convert_connector` match arm
   - Config files: Adds base URL to `development.toml`, `sandbox.toml`, `production.toml`
   - Field probe: Adds match arm

3. **Auto-detects all flows** by parsing `ConnectorServiceTrait` bounds and generates empty impls for every one.

4. **Validates** by running `cargo check` after generation.

5. **Has rollback** — on any error, it restores all backed-up files.

## Phase 4: Determine Implementable Flows & Implementation Order

### 4.1: Determine Implementable Flows

For EVERY flow found in the techspec (Phase 0), assign a status:

| Status | Meaning |
|--------|---------|
| **PLAN** | Flow needs implementation — techspec has sufficient info |
| **EXISTING** | Flow already implemented in connector code — skip |
| **NOT_SUPPORTED** | Techspec lacks info for this flow (no endpoint, no schema) — skip with reason |

A flow is **NOT_SUPPORTED** if:
- The techspec has no section covering that flow's API endpoint
- The connector does not offer that capability (e.g., no void/cancel endpoint)
- The techspec explicitly states the feature is unavailable

A flow has **sufficient info** if the techspec provides at minimum:
- An API endpoint (URL + HTTP method)
- A request body schema (for POST flows) or URL parameter pattern (for GET flows)
- A response body schema with status field

**Exception for mandatory payment methods**: If `{MANDATORY_PAYMENT_METHODS}` requires a payment method (e.g., UPI) and that PM requires certain flows (e.g., Authorize must handle UPI), then the Authorize flow is PLAN even if the techspec only partially documents UPI. The Flow Agent will use pattern guides + similar connector references to fill gaps. Mandatory PMs drive flow inclusion — techspec completeness is secondary for mandatory items.

**Important**: Do NOT limit analysis to only 6 flows. If the techspec documents CreateOrder, SetupMandate, IncomingWebhook, 3DS flows, or any other capability, include them in the plan.

### 4.2: Apply Dependency Graph for Implementation Order

Apply the dependency graph. The order MUST respect these tiers — flows within a tier can be ordered based on techspec analysis, but tiers must be sequential:

```
TIER 0 — Pre-Authorize Setup (no dependencies, come BEFORE Authorize):
  CreateOrder, CreateAccessToken, CreateConnectorCustomer,
  SessionToken, SdkSessionToken, PaymentMethodToken, PreAuthenticate

TIER 1 — Authorization:
  Authorize (depends on: Tier 0 flows if any are present)

TIER 2 — Post-Authorize Authentication (if 3DS):
  Authenticate (depends on: PreAuthenticate)
  PostAuthenticate (depends on: Authenticate)

TIER 3 — Payment Lifecycle (depends on: Authorize):
  PSync (needs connector_transaction_id from Authorize)
  Capture (testing agent runs its own MANUAL-capture Authorize)
  IncrementalAuthorization (needs connector_transaction_id from Authorize)

TIER 4 — Refund (depends on: a CHARGED payment):
  Refund (needs connector_transaction_id from auto-captured Authorize)

TIER 5 — Refund Sync (depends on: Refund):
  RSync (needs connector_refund_id from Refund)

TIER 6 — Void / Reversal (independent test — own Authorize):
  Void (testing agent runs its own MANUAL-capture Authorize, NOT captured)
  VoidPC (testing agent runs its own Authorize+Capture, then reverses)

TIER 7 — Recurring / Mandate (depends on: Authorize):
  SetupMandate (needs connector_transaction_id from Authorize)
  RepeatPayment (needs mandate from SetupMandate)
  MandateRevoke (needs mandate from SetupMandate)

TIER 8 — Dispute (depends on: a charged payment):
  DSync, AcceptDispute, SubmitEvidence, DefendDispute

TIER 9 — Webhooks (typically last, no build dependency):
  IncomingWebhook
```

**Only include flows with status PLAN or EXISTING_PM_PENDING in the implementation order.** EXISTING (with all mandatory PMs covered) and NOT_SUPPORTED flows are listed in the summary but not in the implementation order.

**If Authorize is EXISTING or NOT_SUPPORTED**, the post-Authorize flows can still be planned IF:
- Authorize is EXISTING: downstream flows can use the existing Authorize implementation for testing prerequisites
- Authorize is NOT_SUPPORTED: ALL flows that depend on Authorize are NOT_SUPPORTED (fatal dependency)

**If there are Tier 0 flows**: They come BEFORE Authorize in the implementation order. The first Flow Agent for a new connector will also create the connector foundation (module, struct, ConnectorCommon).

### 4.2.1: Detect Pending PM Additions to EXISTING Flows

When a flow (typically Authorize) has status EXISTING, check whether ALL mandatory payment methods from `{MANDATORY_PAYMENT_METHODS}` are already implemented in that flow.

**Detection process:**
1. Read the connector's transformers file and identify which `PaymentMethodData` variants have match arms in the `TryFrom` impl for the authorize request type
2. Compare against `{MANDATORY_PAYMENT_METHODS}` — for each mandatory PM key (e.g., `"UPI"`, `"WALLET"`, `"NET BANKING"`), check if the corresponding `PaymentMethodData` variant is handled
3. Any mandatory PM that does NOT have a match arm is a **pending PM addition**

**If pending PM additions exist**, change the flow's status from EXISTING to **EXISTING_PM_PENDING** and include it in the implementation order:

| Status | Meaning |
|--------|---------|
| **EXISTING_PM_PENDING** | Flow is implemented but missing mandatory payment methods. Flow Agent must ADD the missing PMs to the existing implementation without rewriting existing PM handling. |

**In the flow plan summary**, list EXISTING_PM_PENDING flows separately:
```
- **Existing (skip)**: [PSync]
- **Existing (PM additions pending)**: [Authorize — has UPI, needs +Wallet, +Netbanking]
```

**In the implementation order**, EXISTING_PM_PENDING flows are placed in Tier 1 (same as Authorize) and include metadata about which PMs are pending:

```
### N. Authorize (EXISTING_PM_PENDING)
- **Status**: EXISTING_PM_PENDING
- **Existing PMs**: [list of already-implemented PMs with PMTs]
- **Pending Mandatory PMs**: [list of missing mandatory PMs with PMTs]
- **Action**: Add match arms for pending PMs to existing TryFrom implementation. Do NOT modify existing PM handling.
- **Sub-Pattern Guides**: [path to each pending PM's sub-pattern guide]
```

**Dependency implication**: If downstream flows (e.g., Capture, Void) require a PM that is pending (e.g., Card for manual capture), the EXISTING_PM_PENDING Authorize must be implemented BEFORE those downstream flows can be tested.
