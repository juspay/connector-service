### How to Detect "Actually Implemented" vs "Stub/Default"

A connector MUST implement all flow traits (Rust's type system requires it), but most are **stubs**. A flow is **actually implemented** if and only if:

1. The connector's `impl ConnectorIntegrationV2<FlowMarker, ...>` block **overrides `get_url()`** to return a non-empty URL string.
2. Alternatively, the flow appears in the connector's `create_all_prerequisites!` macro invocation with a `request_body` and/or `response_body` — this means the connector has real request/response transformers for that flow.

A flow is a **stub** if:
- The `impl ConnectorIntegrationV2<FlowMarker, ...> for ConnectorName<T> {}` block is **empty** (no method overrides).
- Or the `get_url()` returns `Ok(String::new())` (empty string).

**Quick heuristic:** Look at the `create_all_prerequisites!` macro call in the connector's `.rs` file. The flows listed in the `api: [...]` array are the actually implemented flows. Everything else is a stub.

## `add_connector.sh` Reference

### Location

`/home/kanikachaudhary/workflow/hyperswitch-prism/grace/rulesbook/codegen/add_connector.sh`

### What It Does

This is the **Hyperswitch Connector Generator v2.0** — a Bash script that scaffolds a new connector's foundation in connector-service. It:

1. **Creates 2 new files:**
   - `crates/integrations/connector-integration/src/connectors/{name}.rs` — Main connector file with struct, `ConnectorCommon`, `BodyDecoding`, and empty stub implementations for ALL flow traits
   - `crates/integrations/connector-integration/src/connectors/{name}/transformers.rs` — Auth type, error response struct, basic payment request/response types with `TryFrom` transformers

2. **Modifies 10+ existing files:**
   - Protobuf: Adds connector enum variant, config message, and oneof entry to `payment.proto`
   - Domain types: Adds to `ConnectorEnum`, gRPC-to-enum mapping, and `ConnectorParams`
   - Router data: Adds `ConnectorSpecificConfig` variant, auth conversion, macro entries
   - Connector module: Adds `pub mod {name}` and re-export
   - Integration types: Adds `convert_connector` match arm
   - Config files: Adds base URL to `development.toml`, `sandbox.toml`, `production.toml`
   - Field probe: Adds match arm

3. **Auto-detects all flows** by parsing `ConnectorServiceTrait` bounds and generates empty impls for every one.

4. **Validates** by running `cargo check` after generation.

5. **Has rollback** — on any error, it restores all backed-up files.

### Usage

```bash
./grace/rulesbook/codegen/add_connector.sh <connector_name_snake_case> <base_url> [--force] [-y]
```

| Argument | Description |
|----------|-------------|
| `connector_name` | Snake_case connector name (e.g., `payu`, `my_gateway`) |
| `base_url` | The connector's API base URL |
| `--force` | Overwrite if connector already exists |
| `-y` / `--yes` | Skip confirmation prompts |

### Examples

```bash
./grace/rulesbook/codegen/add_connector.sh stripe https://api.stripe.com/v1
./grace/rulesbook/codegen/add_connector.sh my_gateway https://api.mygateway.com --force -y
```

## Phase 4: Determine Implementable Flows & Implementation Order

### 4.1: Determine Implementable Flows (6 core flows only)

For each of the 6 core flows, assign a status:

| Status | Meaning |
|--------|---------|
| **PLAN** | Flow needs implementation — techspec has sufficient info |
| **EXISTING** | Flow already implemented in connector code — skip |
| **EXISTING_PM_PENDING** | Flow is implemented but missing mandatory payment methods — needs PM additions |
| **NOT_SUPPORTED** | Techspec lacks info for this flow (no endpoint, no schema) — skip with reason |

A flow has **sufficient info** if the techspec provides at minimum:
- An API endpoint (URL + HTTP method)
- A request body schema (for POST flows) or URL parameter pattern (for GET flows)
- A response body schema with status field

**Exception for mandatory payment methods**: If `{MANDATORY_PAYMENT_METHODS}` requires a payment method and that PM requires the Authorize flow, then Authorize is PLAN even if the techspec only partially documents it. The Flow Agent will use pattern guides + similar connector references to fill gaps.

### 4.2: Apply Dependency Graph for Implementation Order

The order MUST respect these tiers:

```
TIER 1 — Authorization:
  Authorize

TIER 2 — Post-Authorize (depends on Authorize):
  PSync, Capture, Void

TIER 3 — Refund (depends on Authorize producing a CHARGED payment):
  Refund

TIER 4 — Refund Sync (depends on Refund):
  RSync
```

**Only include flows with status PLAN or EXISTING_PM_PENDING in the implementation order.** EXISTING and NOT_SUPPORTED flows are listed in the summary but not in the implementation order.

**If Authorize is EXISTING or NOT_SUPPORTED**:
- Authorize is EXISTING: downstream flows can use the existing implementation
- Authorize is NOT_SUPPORTED: ALL downstream flows are NOT_SUPPORTED (fatal dependency)

### 4.2.1: Detect Pending PM Additions to EXISTING Flows

When a flow (typically Authorize) has status EXISTING, check whether ALL mandatory payment methods from `{MANDATORY_PAYMENT_METHODS}` are already implemented.

**Detection process:**
1. Read the connector's transformers file and identify which `PaymentMethodData` variants have match arms in the `TryFrom` impl for the authorize request type
2. Compare against `{MANDATORY_PAYMENT_METHODS}` — for each mandatory PM key, check if the corresponding `PaymentMethodData` variant is handled
3. Any mandatory PM that does NOT have a match arm is a **pending PM addition**

**If pending PM additions exist**, change the flow's status from EXISTING to **EXISTING_PM_PENDING** and include it in the implementation order:

**In the flow plan summary**, list EXISTING_PM_PENDING flows separately:
```
- **Existing (skip)**: [PSync]
- **Existing (PM additions pending)**: [Authorize — has UPI, needs +Wallet, +Netbanking]
```

**In the implementation order**, EXISTING_PM_PENDING flows include metadata about which PMs are pending:

```
### N. Authorize (EXISTING_PM_PENDING)
- **Status**: EXISTING_PM_PENDING
- **Existing PMs**: [list of already-implemented PMs with PMTs]
- **Pending Mandatory PMs**: [list of missing mandatory PMs with PMTs]
- **Action**: Add match arms for pending PMs to existing TryFrom implementation. Do NOT modify existing PM handling.
- **Sub-Pattern Guides**: [path to each pending PM's sub-pattern guide]
```
